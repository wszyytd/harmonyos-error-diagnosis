## 3. Debug 签名验证

### 3.1 原始状态

根 `build-profile.json5` 的 `signingConfigs` 为空。直接执行 `assembleHap` 成功，但 Hvigor 明确输出：

```text
Will skip sign 'hos_hap'. No signingConfigs profile is configured in current project.
```

原始产物为：

```text
entry/build/default/outputs/default/entry-default-unsigned.hap
```

因此，设备中已有 `debug: true` 的应用不能反向证明该本地产物已经签名。

### 3.2 本轮签名方式与边界

为只验证运行闭环，本轮使用当前 SDK 随附的 OpenHarmony **公开测试签名材料**，在 `entry/build/t0.5-signing/` 中临时生成：

- 绑定当前模拟器 UDID、有效期 30 天的 Debug profile；
- 测试应用证书链；
- Debug signed HAP；
- `verify-app` 的证书链与 profile 验证输出。

这不是正式发布证书，也不是用户/组织开发者证书。虽然 SDK 测试证书的 subject 含 `Application Release`，HAP 的安装类型由 provision profile 决定；本轮 profile 类型经工具直接验证为 `debug`。本轮未把测试材料复制到源码目录，未在 `build-profile.json5` 中写入口令，也未配置正式签名。

签名与验签使用 `hap-sign-tool.jar`。命令中的 SDK 测试密钥口令不写入报告、配置或源码，以占位符表示：

```powershell
java -jar hap-sign-tool.jar sign-profile `
  -keyAlias '<SDK debug profile alias>' `
  -mode localSign -profileCertFile '<SDK debug profile cert>' `
  -inFile '<temporary debug profile json>' `
  -keystoreFile '<SDK OpenHarmony test keystore>' `
  -keyPwd '<temporary SDK test password>' `
  -keystorePwd '<temporary SDK test password>' `
  -outFile '<ignored build path>/debug-profile.p7b'

java -jar hap-sign-tool.jar sign-app `
  -mode localSign -profileFile '<ignored build path>/debug-profile.p7b' `
  -inFile 'entry-default-unsigned.hap' `
  -outFile '<ignored build path>/entry-default-final-debug-signed.hap' `
  '<other SDK test certificate parameters omitted>'

java -jar hap-sign-tool.jar verify-app `
  -inFile '<ignored build path>/entry-default-final-debug-signed.hap' `
  -outCertChain '<ignored build path>/final-verified-cert-chain.cer' `
  -outProfile '<ignored build path>/final-verified-profile.p7b'
```

验签关键输出：

```text
Find Hap Signing Block success, version: 3, block count: 3
profile type is: debug
verify codesign success
Digest verify result: true, DigestAlgorithm: SHA-256
verify: Verify success
verify-app success
```

`bm dump` 的 `appSignType` 仍显示 `none`，该字段没有反映本地 OpenHarmony 测试签名类型，不能替代 HAP 文件的直接验签结果。本轮以 `verify-app`、Debug profile 解包结果和安装结果作为签名证据。

## 4. 构建、安装与启动

### 4.1 构建

```powershell

& 'D:\Dev\IDE\DevEco Studio\tools\hvigor\bin\hvigorw.bat' `
  --mode module `
```

最终复验：`BUILD SUCCESSFUL in 12 s 225 ms`。ArkTS 编译、资源编译和 HAP 打包均成功。Hvigor 内置签名仍按预期跳过，随后由隔离的 T0.5 步骤生成 Debug signed HAP。

### 4.2 安装

首次执行覆盖安装时返回：

```text
code:9568332 error: install sign info inconsistent
```

原因是 DevEco 手动安装的旧验证包与本轮 SDK Debug 测试签名不一致。仅卸载 `com.example.myapplication` 这个无业务数据的模板应用后重新安装：

```powershell
hdc -t 127.0.0.1:5555 uninstall com.example.myapplication
hdc -t 127.0.0.1:5555 install '<ignored build path>/entry-default-debug-signed.hap'
```

结果：

```text
uninstall bundle successfully
install bundle successfully
```

后续同一签名的探针/清理版本均可使用 `install -r` 成功覆盖。

关键结果：


清空日志、启动并等待 5 秒后，应用 PID 仍存在。使用 `hilog -x -P <pid>` 检索 `FATAL`、`SIGABRT`、`SIGSEGV`、`AppFreeze`、`crash`、`uncaught`、`exception`，没有匹配到阻塞性运行错误。

## 5. Core Vision OCR 实测

### 5.1 测试输入

使用本地生成的固定 PNG，白底黑字，包含：

```text
ArkTS ERROR: Property 'repoUrl' has no initializer.
Hvigor ERROR: assembleHap failed.
OHPM ERROR: dependency @ohos/hypium not found.
```

图片是合成测试夹具，不包含用户隐私、Token、账号或签名信息。源图片存放在 `entry/build/t0.5-runtime/`；临时复制到 `entry/src/main/resources/rawfile/` 只为运行探针，验证后已删除，未进入最终源码 diff。

### 5.2 探针调用

探针仅临时加入模板 `EntryAbility`，完成以下最小步骤：

1. 使用 `resourceManager.getRawFd()` 读取固定 PNG；
2. 使用 Image Kit 解码为 RGBA_8888 `PixelMap`；
3. 调用 `textRecognition.init()`；
4. 若初始化成功，再调用 `recognizeText({ pixelMap })` 并记录 `result.value`；
5. 释放 PixelMap、ImageSource 与 OCR service。

探针不包含页面、网络、存储或诊断业务逻辑。

### 5.3 实际结果

第一次从 `/data/local/tmp` 直接创建 `ImageSource` 时，返回：

```text
Cannot read property createPixelMap of undefined
```

该失败属于应用无法用该 shell 路径创建图像源，不是 OCR 结论。改用应用 `rawfile` 后，图像解码成功，日志推进至 OCR 初始化：

```text
T0_OCR_STEP=createPixelMap
T0_OCR_STEP=init
T0_OCR_ERROR code= message=Cannot read property init of undefined
```

结论：

- `@kit.CoreVisionKit` 与 `@hms.ai.ocr.textRecognition.d.ts` 在 SDK 中存在，探针可通过 ArkTS 编译；
- 固定 PNG 和 RGBA_8888 `PixelMap` 解码成功；
- 当前模拟器运行时的 `textRecognition` 对象为 `undefined`，因此没有真正进入 `init()`，也没有调用 `recognizeText()`；
- 没有 OCR 返回文本；
- 失败早于 OCR API 的权限/业务校验，不是相册权限或 OCR 权限问题；
- 本地声明未要求 OCR 本身的运行时权限；图片访问权限仍取决于未来正式图片来源；
- 当前模拟器不能作为 OCR 支持设备；必须在支持 Core Vision 的 HarmonyOS 真机或课程认可的远程真机复测。

### 5.4 降级与任务影响

- T20 必须保留手动输入/粘贴作为降级，并在运行时能力不可用时直接回退；
- 不得生成假识别文本，不得把当前模拟器写成 OCR 支持；
- OCR 失败不影响构建、安装、启动、手动输入、本地规则诊断、保存与历史查看；
- 因此该问题阻塞 T20 和最终 OCR 验收，不阻塞 T1；
- Form Kit、Natural Language Kit、Data Augmentation Kit、沉浸光感仍只有 T0 静态证据，其设备运行验证属于 P1，不阻塞 P0 工程骨架。
