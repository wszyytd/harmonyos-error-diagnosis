# T0 能力兼容矩阵

## 1. 判定规则

- **PASS（编译）**：本机 SDK 中存在准确声明，并已在当前 API 26 Stage 工程完成导入编译。
- **PASS（运行）**：已在本次验证的设备上实际调用并得到预期结果。
- **UNVERIFIED**：存在静态证据，但没有本次设备运行证据。
- **UNAVAILABLE**：本地声明或设备能力表明确不适用于目标设备。
- **BLOCKED**：具备理论验证路径，但被设备、签名或环境问题阻塞。

“SDK 中存在”不等于“当前模拟器或真机运行支持”。本矩阵分别记录这两个维度。

## 2. 总览

| 能力 | SDK/API 存在 | 导入编译 | phone 静态能力 | 本机模拟器运行 | 真机运行 | 当前项目可直接采用 |
|---|---|---|---|---|---|---|
| Core Vision OCR | 是 | PASS | 能力表包含 OCR | FAIL / UNAVAILABLE | UNVERIFIED | 只能以适配器+手动输入降级接入；T20 等待真机证据 |
| Form Kit | 是 | PASS | 能力表包含 Form | UNVERIFIED | UNVERIFIED | P1 可编码，桌面宿主和刷新/拉起需设备验证 |
| Natural Language Kit | 是 | PASS | 能力表包含 NLP | UNVERIFIED | UNVERIFIED | P1 可选增强，必须保留本地关键词回退 |
| Data Augmentation Kit | 是 | PASS，有系统能力警告 | Retrieval/部分能力存在，RAG 不在 phone | UNVERIFIED；完整 RAG 不适用 phone | UNVERIFIED | phone 不可直接采用完整 RAG；只可分能力检测并回退 |
| 沉浸光感 | 是 | PASS | ArkUI Full，API 26 | UNVERIFIED | UNVERIFIED | P1 可使用 `uiMaterial`，必须按算力降级 |

## 3. Core Vision OCR

### 3.1 本地证据

| 项目 | 结论 |
|---|---|
| Kit | `@kit.CoreVisionKit` |
| 导出符号 | `textRecognition` |
| 底层声明 | `@hms.ai.ocr.textRecognition.d.ts` |
| 系统能力 | `SystemCapability.AI.OCR.TextRecognition` |
| API 起始版本 | `recognizeText` 自 4.0.0(10)；`init/release` 自 5.0.0(12) |
| 设备能力表 | phone/tablet 自 5.0.0(12)，2in1 自 5.0.1(13) |
| 输入 | `image.PixelMap`；本机离线 API 文档注明目前仅支持 RGBA_8888 |
| 输出 | `TextRecognitionResult.value` 与 blocks/lines/words 坐标信息 |
| 支持语言 | 本机离线文档列出简中、英文、日文、韩文、繁中 |
| 额外 OHPM 依赖 | 无，随 HarmonyOS SDK 提供 |
| Stage 约束 | 是 |
| 导入编译 | PASS |

本机声明路径：

```text
sdk/default/hms/ets/kits/@kit.CoreVisionKit.d.ts
sdk/default/hms/ets/api/@hms.ai.ocr.textRecognition.d.ts
sdk/default/hms/ets/api/device-define/syscap-version.json
plugins/openharmony/ohos-info-center-view/static/hos/JsEtsAPIReference/zh-cn_topic_0000002579222180.html
```

### 3.2 权限与签名

- `textRecognition` 声明中没有 `@permission`，因此 **OCR API 本身未声明运行时权限**。
- 图片来源应优先使用系统图片选择器，只处理用户主动选择的一张图片及其临时 URI；OCR Service 再将图片解码为 `PixelMap`。
- 若绕过系统选择器直接遍历或读取媒体库，则相关 PhotoAccess API 可要求 `ohos.permission.READ_IMAGEVIDEO`；本项目不应采用该扩大权限的路径。
- OCR 不要求额外 OHPM 包，但应用安装本身仍需要有效调试/发布签名。
- 是否存在面向特定开发者或设备的能力开放申请，本地 SDK 未给出额外配置证据，标记为**待官方控制台/真机确认**。

### 3.3 模拟器与真机

- 本地 SDK 设备能力表表明 phone/tablet/2in1 类型具有 OCR 系统能力；这只是静态目标设备定义。
- T0.5 中 Pura 90 Pro API 26 x86_64 模拟器已通过 HDC 连接，Debug signed HAP 已安装和启动。
- 固定合成错误图片从应用 `rawfile` 解码为 RGBA_8888 `PixelMap` 成功，说明输入与图像解码链路可用。
- 运行至 OCR 初始化阶段时，`@kit.CoreVisionKit` 的 `textRecognition` 导出对象为 `undefined`，错误为 `Cannot read property init of undefined`。失败早于 `init()` 调用，没有 OCR 业务错误码，不是运行时权限拒绝。
- 因此当前模拟器明确不能作为 OCR 支持设备；SDK 声明可编译不能替代运行时能力证据。
- 当前没有真机或远程真机证据。
- 结论：当前模拟器 **FAIL / UNAVAILABLE**，真机 **UNVERIFIED**，不得写成“支持”。是否为所有模拟器镜像限制仍需官方说明；项目验收应以支持 Core Vision 的 HarmonyOS 真机或课程认可的远程真机为准。

### 3.4 采用与降级

当前项目可以编译 OCR 适配器，但在至少一个设备返回真实英文错误日志文本之前，T20 仍是阻塞项；这不阻塞 T1～T19 中不依赖 OCR 的工程骨架、手动输入与本地诊断链路。降级方案：

1. 图片选择取消、OCR 初始化失败、超时、无文字或能力缺失时停止加载；
2. 不生成假文本；
3. 保留并突出手动输入/粘贴入口；
4. OCR 不可用不得阻止本地规则诊断、保存和历史查看。

## 6. Data Augmentation Kit

### 6.1 本地证据

| 项目 | 结论 |
|---|---|
| Kit | `@kit.DataAugmentationKit` |
| API 起始版本 | 6.0.0(20) |
| 导出 | `retrieval`、`rag`、`knowledgeProcessor`、`localChatModel` |
| Stage | `rag` 声明为 Stage only |

| 额外 OHPM 依赖 | 无 |
| 权限 | Kit 顶层声明未给出普通用户权限；数据源/数据库访问仍受应用沙箱约束 |

### 6.2 设备差异

本机 `syscap-version.json` 给出的静态矩阵：

| 子能力 | phone | tablet | 2in1 |
|---|---|---|---|
| Retrieval | 自 API 20 | 自 API 20 | 自 API 20 |
| KnowledgeProcessor | 自 API 20 | 自 API 20 | 自 API 20 |
| RAG | 不在 phone 能力表 | 不在 tablet 能力表 | 自 API 20 |
| LocalChatModel | 不在 phone 能力表 | 不在 tablet 能力表 | 自 API 20 |

同时，`@hms.data.knowledgeProcessor.d.ts` 顶部明确写明其依赖的 embedding 模型只能部署在 PC/2in1，因此知识处理能力仅支持 PC/2in1。该描述与通用 syscap-version 表中列出 phone/tablet 存在口径冲突。

处理原则：以更严格的 API 声明约束和运行时能力检测为准，不能仅凭 syscap 表宣称 phone 可用。

### 6.3 采用结论

- 当前项目主目标是 phone，**不能直接采用完整 RAG 或 LocalChatModel**。
- `retrieval` 是否可在目标手机用于项目知识库仍需真机最小运行探针。
- P1 最小落地方案应先使用应用内只读 JSON/关系数据库与本地规则检索；Data Augmentation 作为可替换增强层。
- 模拟器和真机运行均为 UNVERIFIED。
- 该能力已统一为 P1，不阻塞 P0。

## 7. 沉浸光感

### 7.1 准确能力名称

需求中的“沉浸光感”在本机 API 26 SDK 中不是一个名为 `ImmersiveLightKit` 的独立 Kit。可直接对应的标准 ArkUI 能力是：

```typescript
import { uiMaterial } from '@kit.ArkUI';

new uiMaterial.ImmersiveMaterial({
  style: uiMaterial.ImmersiveStyle.ULTRA_THICK,
  interactive: true,
  lightEffect: { color: Color.White }
})
```

准确声明位于 `@ohos.arkui.uiMaterial.d.ts`：

- `uiMaterial.ImmersiveMaterial`
- `uiMaterial.ImmersiveOptions`
- `uiMaterial.LightEffectOptions`
- `ImmersiveOptions.lightEffect`

### 7.2 兼容性

| 项目 | 结论 |
|---|---|
| 最低 API | 26.0.0 |
| 系统能力 | `SystemCapability.ArkUI.ArkUI.Full` |
| 模型 | Stage only |
| 设备声明 | 离线 API 文档标注 phone、2in1、tablet、wearable、tv |
| OHPM 依赖 | 无 |
| 权限 | 无 |
| 导入编译 | PASS |
| 视觉运行 | UNVERIFIED |

离线官方 API 文档说明材质效果会随设备算力高/中/低档变化，部分反色和阴影效果只在高档或中档设备生效。因此“可编译”不代表所有设备视觉一致。

降级方案：

1. 能力不可用或低算力时使用标准背景、边框、阴影和按压态；
2. 深色模式、减少动态效果或对比度不足时减弱/关闭 lightEffect；
3. 不把光感作为状态识别的唯一手段；
4. 不让材质初始化影响主要操作响应。
