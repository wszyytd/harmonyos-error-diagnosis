# 开发报错诊断助手视觉与动效设计说明书

> 版本：1.0（设计稿）
> 设计方向：A「光感玻璃焦点」
> 适用阶段：P0 稳定基线完成后的 P1 视觉增强
> 验证状态：参数均为建议范围；`uiMaterial` 已通过 API 26 编译探针，视觉效果尚未完成设备验收

## 1. 目的、范围与设计原则

本文为“开发报错诊断助手”规定一套简洁、可实现、便于课程答辩展示的视觉与动效方案，适用于首页、工具页、记录页、设置页、诊断页、报告页，以及后续诊断完成、保存成功和重试等状态反馈。

本文不改变现有业务流程、页面路由、ViewModel、Repository 或本地诊断架构，也不替代需求规格说明书、概要设计说明书和详细设计说明书。视觉增强属于 P1，不得打乱 T17～T24 的 P0 开发顺序。

设计目标与约束如下：

- 用一个明确的视觉焦点强调核心诊断操作；
- 保证错误日志、原因和排查步骤清晰、稳定、易读；
- 使用 ArkUI 系统材质和 HarmonyOS Symbol 体现平台特征；
- 页面内容区域最多一个强光焦点，不进行全屏玻璃化；全局悬浮 Dock 使用统一低强度玻璃光感，不与页面内容争夺层级；
- 玻璃材质只用于关键入口、主操作或结论区，普通内容继续使用稳定实色表面；
- 不使用装饰性无限动画，动效必须可中断、可关闭、可降级；
- 不以颜色、透明度、模糊或动画作为唯一状态表达；
- 错误日志和报告正文后方不放置复杂动态背景；
- LIGHT、DARK、SYSTEM 使用同一套语义 Token；SYSTEM 采用系统解析后的有效主题；
- 正文对比度目标不低于 4.5:1，关键点击区域不小于 44vp × 44vp；
- 系统字体放大至 1.5 倍时，主要文字和操作不得被截断到无法理解。

## 2. 视觉 Token

### 2.1 与 T12 的关系

现有 `ThemePalette` 已提供 `pageBackground`、`surfaceBackground`、`navigationBackground`、`primaryText`、`secondaryText`、`accent` 和 `error`。视觉增强应扩展这份语义 Palette 或建立由它计算出的只读视觉 Token，不得在页面中散落另一套颜色常量。

现有颜色保持不变：

| Token | LIGHT | DARK |
|---|---|---|
| 页面背景 `pageBackground` | `#F6F7FB` | `#101216` |
| 普通卡片 `surfaceBackground` | `#FFFFFF` | `#1A1D23` |
| 主文字 `primaryText` | `#171A1F` | `#F3F5F8` |
| 次级文字 `secondaryText` | `#5D6673` | `#B5BDC9` |
| 强调色 `accent` | `#0A59F7` | `#7EACFF` |
| 错误色 `error` | `#B3261E` | `#FFB4AB` |

### 2.2 建议扩展 Token

以下为实现前的建议范围，不代表已在模拟器或真机完成视觉验收：

| 语义 | LIGHT 建议 | DARK 建议 | 使用限制 |
|---|---|---|---|
| 成功色 | `#137A4B` 附近 | `#63D7A0` 附近 | 同时配成功文字或图标 |
| 警告色 | `#8A5A00` 附近 | `#FFD166` 附近 | 同时配“警告”文字 |
| 玻璃焦点底色 | `accent` 的 8%～14% 叠加 `surfaceBackground` | `accent` 的 12%～20% 叠加 `surfaceBackground` | 不直接放在长正文后方 |
| 玻璃边框 | `accent` 的 20%～30% | `accent` 的 28%～40% | 建议 1vp |
| 光晕色 | `accent` 的 12%～20% | `accent` 的 16%～26% | 页面内容最多一处；Dock 只允许选中项使用小范围光晕 |
| 遮罩 | 黑色 4%～8% | 黑色 8%～14% | 仅用于保证叠层分离 |
| 背景模糊 | 16～24vp | 12～20vp | 降低效果档改为 8～12vp |

颜色选择应先验证文字对比度；若玻璃背景导致对比度不足，应提高底色不透明度或直接降级为实色卡片，而不是增加文字阴影。

## 3. 玻璃焦点统一样式

玻璃焦点是一种视觉样式，不承担业务判断。建议以公共 Modifier 或轻量包装组件管理；主题数据来自扩展后的 Token，业务状态仍由页面或 ViewModel 输入。

| 项目 | 建议范围 |
|---|---|
| 圆角 | 卡片 16～20vp；按钮 14～18vp |
| 边框 | 1vp 半透明强调色 |
| 内边距 | 卡片 16～20vp；按钮水平 20vp、垂直不低于 12vp |
| 光晕 | 单层、低透明度，扩散范围约 12～24vp |
| 背景模糊 | 完整效果 16～24vp；降低效果 8～12vp |
| 文字层级 | 主标题使用 `primaryText`；说明使用 `secondaryText`；不能使用半透明文字模拟层级 |
| 图标层级 | 图标辅助语义，不替代标题和状态文字 |

状态规则：

- 正常：显示玻璃背景、边框和单层弱光晕；
- 按下：缩放至约 0.98，光晕略减弱，避免产生漂浮失控感；
- 禁用：取消光晕并降低材质强调，同时保留明确禁用文案或状态；
- 加载：保留静态玻璃外观，加载进度继续复用 T13 的 `LoadingProgress` 和文字，不新增循环扫光；
- 错误：玻璃区域不整体变成强红色，错误由 `InlineErrorNotice`、错误文字和图标表达；
- DARK：降低大面积白色反光，允许略强的边框，但必须保持正文对比度；
- 能力不可用：使用 `surfaceBackground`、1vp 边框和标准按压反馈，布局、尺寸和点击行为不变。

## 4. 页面设计

### 4.1 悬浮发光玻璃 Dock

底部 Dock 是本方案的确定设计，不是可选装饰。正式实现必须同时满足“悬浮”和“发光玻璃”两个特征：Dock 与屏幕左右、底部安全区之间保留可见间距，覆盖在页面内容之上，不得做成紧贴屏幕底边的普通矩形导航栏。

#### 4.1.1 导航结构

Dock 包含四个一级入口：

| 导航项 | 对应页面 | 主要内容 |
|---|---|---|
| 首页 | `HomePage` | 截图诊断、文本诊断、最近记录摘要 |
| 工具 | `ToolsPage` | JSON 验证、HTTP 状态码解释 |
| 记录 | 历史记录页面 | 已保存报告、删除和清空入口 |
| 设置 | 设置页面 | 主题、主题色、字号、光感等级和隐私入口 |

诊断页和报告页属于任务流程页面，不作为 Dock 项。用户从首页进入诊断页或从记录页打开报告时，流程页面压栈显示并隐藏 Dock；返回一级页面后恢复 Dock。隐私页面仍归入设置页面组，不成为独立 Dock 项。

#### 4.1.2 布局和材质

| 项目 | 设计要求 |
|---|---|
| 定位 | 覆盖在一级页面内容上方，水平居中，必须呈悬浮状态 |
| 左右间距 | 建议 16～20vp |
| 底部间距 | 位于系统安全区上方约 12～16vp；实际值由安全区计算 |
| 高度 | 建议 64～72vp；1.5 倍字体下允许增加高度 |
| 圆角 | 建议 24～30vp |
| 页面预留 | 一级页面滚动内容增加约 96～112vp 底部空间，最后一项不得被 Dock 遮挡 |
| 玻璃材质 | 优先使用 API 26 `uiMaterial.ImmersiveMaterial`，建议背景模糊 16～24vp |
| 发光 | Dock 外沿使用单层弱光晕；当前选中项使用主题色小范围光晕 |
| 边框 | 1vp 半透明亮边，LIGHT 和 DARK 分别调整不透明度 |

Dock 容器的玻璃效果应在页面滚动时保持稳定，不能随内容产生高频闪烁。选中项同时使用图标、文字、主题色和轻量背景表达，不能只依赖发光。未选中项保持较低对比层级，但文字仍须可读。

#### 4.1.3 交互状态

- 初次出现：Dock 整体进行一次 6vp 以内上移和淡入，总时长 200～260ms；
- 切换导航：选中背景在 160～220ms 内移动或交叉淡入，Symbol 可播放一次已验证的系统效果；
- 按下：当前项目缩放至约 0.96～0.98，释放后回弹；
- 重复点击当前项：不重复播放强光动画，可选择将当前页面滚动到顶部；
- 页面切换：保留各一级页面的必要局部状态，不能重复压入同一个顶层路由；
- 进入诊断或报告：Dock 隐藏，避免遮挡编辑器、报告操作和系统返回手势；
- 正式应用禁止 Dock 持续呼吸、无限扫光或自动循环发光。

#### 4.1.4 强制降级规则

能力不可用时允许降低模糊、取消外部光晕或改用实色半透明背景，但 Dock 的悬浮位置、圆角轮廓、四项导航、选中状态和安全区间距必须保留。换言之，材质可以降级，Dock 不能降级成贴底普通导航条，也不能因为光感失败而消失。

### 4.2 首页

当前 `HomePage` 包含截图诊断、文本诊断、验证工具、最近记录和主题验证区域。视觉增强不得重排这些业务入口。

- 唯一强光焦点：**文本诊断入口**。它是当前环境中最稳定、最适合答辩演示的核心闭环；
- 截图诊断：保留普通主入口卡片样式。待 T20 OCR 真机能力闭环后，可增加小型“截图识别”标识，但不与文本诊断争夺强光焦点；
- 验证工具：保持普通按钮或次级卡片；
- 最近记录：继续使用 `HistoryReportCard` 的普通实色表面，不使用玻璃或强光；
- 最近记录的加载、空数据和失败：继续复用 T13 的 `AsyncStatePanel`、`EmptyStateView` 和 `InlineErrorNotice`；
- 页面首次进入时，文本诊断玻璃卡可进行一次轻微淡入和 8vp 以内上移；从子页面返回时不重复播放；
- 页面保持可滚动。1.5 倍字体时允许入口卡片自然增高，不固定标题或说明的单行高度；
- 静态降级时，文本诊断入口改为带强调色边框的普通卡片，仍保持首页第一视觉层级。

悬浮 Dock 使用低强度全局玻璃层级；首页内容区仍只让文本诊断入口成为强光焦点。截图诊断和文本诊断不得同时持续发光。

### 4.3 诊断页

当前 `DiagnosisPage` 包含标题、IMAGE 降级提示、文本编辑器、诊断状态、示例按钮、开始诊断和返回按钮。

- 唯一强光焦点：**“开始诊断”按钮**；
- `DiagnosisTextEditor` 保持实色背景和清晰边框，不玻璃化；
- 正常：按钮显示玻璃材质、明确文字和可点击状态；
- 按下：缩放到 0.98 左右，按下约 90～120ms，释放回弹约 160～220ms；
- 加载：按钮禁止重复提交，取消按压动画和强光变化；加载状态继续由 T13 组件显示进度与“正在生成诊断报告”等文字；
- 失败：按钮恢复可点击，错误提示使用 `InlineErrorNotice`；重试可触发一次 Symbol 反馈，但不将错误提示玻璃化；
- 禁用：无输入或正在处理时不显示强光晕，并提供可理解的输入校验或加载文字；
- IMAGE 手动降级提示保持 INFO 层级，弱于主按钮；
- 长文本和键盘出现时，页面保持滚动，编辑器和开始按钮均可到达；不得用固定底部浮层遮挡编辑区域；
- 减少动态时只保留标准按下态，或完全取消缩放。

### 4.4 报告页

当前 `ReportPage` 已按错误类型与摘要、证据、原因、步骤、建议、风险、原始文本和操作区组织内容。

- 唯一强光焦点：**错误类型与结论头部区域**；
- evidence、原因、步骤、建议继续使用 `ReportSectionCard` 普通实色表面；
- 风险提示必须同时包含“风险提示”文字、警告图标和清晰边框，不只依靠警告色；
- 结论头部先显示，内容区随后以短距离淡入。所有区块应在约 360ms 内可读，不等待动画结束才能滚动或操作；
- 原始文本区域使用稳定实色背景，不透明玻璃，不降低日志可读性；
- 长报告保持滚动，动画不改变最终布局高度，避免内容跳动；
- T18 增加保存按钮后，保存按钮保持普通主要操作样式。保存成功使用短暂 Symbol 和文字反馈，不增加第二个强光焦点；
- 从历史记录打开已有报告时，可只播放结论区淡入，不重复播放“诊断完成”动效；
- 静态降级时，结论区使用强调色细边框的普通卡片。

### 4.5 设置页

设置页是 Dock 的一级页面，页面本身使用普通实色或低透明度卡片，不再增加第二个强光内容焦点。Dock 中“设置”选中项承担该页的光感导航反馈。

课程设计只提供以下必要设置：

| 设置项 | 选项 | 实现原则 |
|---|---|---|
| 主题模式 | SYSTEM、LIGHT、DARK | 复用 T12 的主题状态和 Preferences 持久化 |
| 主题色 | 蓝、青、紫、绿四个预设 | 不提供任意取色器；每个预设需检查 LIGHT/DARK 对比度 |
| 内容字号 | 紧凑、标准、大号 | 使用离散档位；仍叠加系统字体缩放，不得绕过系统无障碍设置 |
| 光感效果 | 完整、降低、关闭 | 控制 Dock、玻璃焦点和非必要动画；关闭后保留悬浮 Dock 轮廓 |

设置变更应即时预览，并通过 `SettingsViewModel → AppStateStore` 更新全局展示；持久化失败时沿用 T12 机制，明确提示“仅本次会话生效”。设置页还提供隐私说明入口，隐私内容继续使用稳定实色背景。

字体设置不使用连续滑块，主题色不使用任意颜色选择器，避免出现难以测试的布局组合和不可读配色。

## 5. Symbol 与动效规范

### 5.1 Symbol 状态语义

具体系统 Symbol 名称和 `symbolEffect` 类型必须在当前 API 26 SDK 中实现前验证。本文只确定语义，不假设某个名称必然存在。

| 状态 | Symbol 语义 | 触发条件 | 建议时间 | 重复规则 | 静态降级与辅助文字 |
|---|---|---|---|---|---|
| 诊断完成 | 完成、对勾或结果就绪 | 本次诊断成功且首次进入新报告 | 280～420ms | 每次新诊断一次 | 静态完成图标 + “诊断完成” |
| 保存成功 | 已保存或对勾 | T18 保存操作首次成功 | 240～360ms | 每次成功操作一次；重复保存不连续播放 | 静态已保存图标 + “报告已保存” |
| 重试 | 刷新、重新执行或回转箭头 | 用户主动点击重试 | 220～320ms | 每次有效点击一次 | 静态重试图标 + 按钮文字 |
| 错误/警告 | 警告、感叹号或错误 | 操作失败或存在风险提示 | 200～300ms，可不播放 | 同一错误不自动反复播放 | 静态警告图标 + 明确错误/风险文字 |

### 5.2 动效参数表

| 页面/组件 | 动效 | 触发 | 属性与数值 | 时间/延迟 | 曲线 | 可中断 | 减少动态降级 |
|---|---|---|---|---|---|---|---|
| 悬浮 Dock | 首次出现 | 一级页面根容器首次可见 | opacity 0→1；Y 6vp→0 | 200～260ms；0 | ease-out | 是 | 直接显示在最终悬浮位置 |
| Dock 选中项 | 导航切换 | 用户选择另一一级页面 | 选中背景交叉淡入；scale 0.96→1 | 160～220ms；0 | 轻量 spring 或 ease-out | 是 | 立即切换图标、文字和静态背景 |
| 首页文本诊断卡 | 焦点出现 | 本次页面首次展示 | opacity 0→1；Y 8vp→0 | 200～260ms；0 | ease-out | 是 | 直接显示 |
| 开始诊断按钮 | 按下 | 手指按下 | scale 1→0.98 | 90～120ms | ease-out | 是 | 标准按下色或无动画 |
| 开始诊断按钮 | 回弹 | 松开且操作有效 | scale 0.98→1 | 160～220ms | 轻量 spring，禁止过冲明显 | 是 | 直接恢复 |
| 诊断完成 Symbol | 单次完成 | 新诊断成功 | scale 0.85→1；opacity 0→1 | 280～420ms；0 | ease-out 或已验证 SymbolEffect | 是 | 静态图标与文字 |
| 报告结论区 | 结论出现 | 报告数据可用 | opacity 0→1；Y 6vp→0 | 180～240ms；0 | ease-out | 是 | 直接显示 |
| 报告内容卡 | 分段淡入 | 新报告首次展示 | opacity 0→1；Y 6vp→0 | 单段 160～220ms；段间 40ms，总计不超过 360ms | ease-out | 是 | 同时直接显示全部 |
| 保存/重试 Symbol | 操作反馈 | 操作成功或有效重试 | 使用验证后的单次 SymbolEffect | 220～360ms；0 | 系统效果 | 是 | 静态图标与文字 |

加载动画只复用已有加载组件并显示状态文字。正式应用不使用无限循环的扫光、霓虹呼吸、背景漂移或装饰旋转。

## 6. 性能、降级与无障碍

### 6.1 三档效果

| 档位 | 启用条件 | 表现 |
|---|---|---|
| 完整效果 | `uiMaterial` 和 Symbol 动效可用、设备表现稳定、未启用减少动态 | Dock 使用完整模糊、单层外沿光晕和选中项光晕；页面保留按压回弹、短时分段淡入、单次 Symbol |
| 降低效果 | 模糊效果较弱、设备掉帧、深色对比度不足或用户偏好更少动态 | Dock 模糊降至 8～12vp并取消外沿光晕；取消分段动画，保留悬浮轮廓、状态文字与基础按压态 |
| 静态降级 | 能力不可用、减少动态开启或验收不稳定 | Dock 使用实色/半透明圆角表面并保持悬浮；静态 Symbol，无缩放、位移和淡入，功能和状态完整保留 |

能力检测属于平台适配层；页面只接收“效果档位”或“是否减少动态”等展示状态，不直接探测设备。具体减少动态和性能能力 API 当前未验证，标记为“实现前验证”。若没有可靠接口，应提供应用内视觉效果开关或在演示设备上选择保守默认值，不能通过猜测设备等级启用效果。

### 6.2 无障碍验收要求

- 正文与背景对比度目标不低于 4.5:1；
- 诊断完成、保存、错误和警告均显示文字，不只使用颜色、光效或 Symbol；
- 所有关键操作区域不小于 44vp × 44vp；
- 1.5 倍字体下允许容器增高、文字换行和页面滚动；
- 动效无快速闪烁，不使用频闪或高对比度连续扫光；
- Symbol 和按钮提供与行为一致的辅助描述；
- 动画期间不抢占焦点，不阻断返回、滚动或重试；
- 动画结束后组件位置、焦点顺序和可操作性保持稳定；
- Dock 四个项目均提供图标、文字和辅助描述，选中状态不能只依赖主题色或光晕；
- 页面最后一个可操作元素在 1.5 倍字体和横屏状态下也不得被悬浮 Dock 遮挡；
- 减少动态时仍可完成全部诊断、报告阅读和保存操作。

## 7. 与现有架构的结合

建议的职责关系如下：

- `AppStateStore`：继续作为 LIGHT/DARK/SYSTEM 及有效主题的单一来源；后续只增加经验证的视觉效果偏好或效果档位，不保存页面动画过程；
- 设置模型与存储：在 `UserSettings`、`SettingsRepository` 和 Preferences 稳定 Key 中增加主题色、字号档位和光感档位；升级时为旧数据提供默认值；
- 主题层：在现有 `ThemePalette` 上扩展成功、警告、玻璃背景、Dock 背景、边框和光晕等语义 Token；
- 一级页面容器：由统一根容器持有四项导航状态和悬浮 Dock；顶层切换不能重复压栈，诊断与报告等流程路由压栈时隐藏 Dock；
- T13 公共状态组件：继续负责加载、空数据、错误和重试表达，不因玻璃效果复制一套状态组件；
- `HomePage`：只消费玻璃样式与首页首次展示状态，不改变 `HomeViewModel` 和最近记录读取；
- `DiagnosisPage`：根据已有 `diagnosisOperation` 控制按钮可用、加载和失败表现；不把动效状态写入诊断 Service；
- `ReportPage`：根据报告是否为本次新诊断决定是否播放一次完成动效；普通报告区继续复用 `ReportSectionCard`；
- T18 保存状态：成功后向展示层提供一次性反馈事件；Repository 不感知动画；
- 平台适配层：封装 `uiMaterial`、Symbol 能力和降级判定，初始化失败不能影响 P0 页面。

推荐只规划一个公共玻璃 Modifier、一个 `FloatingGlassDock`、一个效果参数模型和必要的 Symbol 状态展示，不为每个页面创建功能相同的玻璃组件。Dock 只接收当前项、四项点击事件、主题 Token、效果档位和安全区数据，不访问 Repository 或业务 Service。本轮不编写这些生产实现。

## 9. 90 秒答辩演示

建议流程控制在 75～90 秒：

1. 0～12 秒：首页展示悬浮发光玻璃 Dock，并说明四项一级导航和安全区设计；
2. 12～25 秒：依次切换工具、记录、设置，展示 Dock 选中项的单次 Symbol/光感反馈；
3. 25～38 秒：在设置页切换深色、主题色或字号，说明设置通过 AppStateStore 和 Preferences 生效；
4. 38～50 秒：返回首页进入文本诊断；流程页隐藏 Dock，日志编辑区保持实色；
5. 50～63 秒：载入固定 ArkTS 示例，点击“开始诊断”，展示轻微按压回弹和文字加载状态；
6. 63～73 秒：诊断完成，展示一次性完成 Symbol；
7. 73～83 秒：展示报告结论玻璃区及证据、原因、步骤，强调正文没有玻璃背景；
8. 83～90 秒：切换降低/关闭光感，证明 Dock 仍保持悬浮且全部功能可用，并说明 API 26、ArkTS、ArkUI 和 `uiMaterial.ImmersiveMaterial`。

备用方案：

- 背景模糊在模拟器不明显：切换静态降级，展示提前保存的同设备对比截图，并如实说明真机待验证；
- Dock 材质初始化失败：立即使用实色圆角悬浮 Dock，不能取消导航或改成贴底布局；
- Symbol 动效不可用：使用静态系统 Symbol、状态文字和普通淡入；
- 演示过程诊断失败：使用固定本地错误样例重试，或直接打开预置/已保存报告；
- 设备掉帧：关闭光晕和分段淡入，只保留三个页面的静态层级设计。

## 10. 风险与实现前验证清单

| 项目 | 当前结论 | 实现前要求 |
|---|---|---|
| `uiMaterial.ImmersiveMaterial` | API 26 类型声明存在，导入编译通过；视觉运行未验证 | 在计划演示模拟器和真机分别截图、录屏并检查帧率 |
| 模拟器/真机差异 | 官方材料效果可能随设备算力变化 | 不要求逐像素一致，真机效果不足时采用降低档 |
| 背景模糊性能 | 未完成目标页面压力测试 | 只允许每页一处，连续诊断和滚动时观察掉帧 |
| Dock 安全区与遮挡 | 尚未在全部手机、平板和横屏尺寸验证 | 使用系统安全区数据；逐页检查最后一个可操作元素和系统手势区域 |
| 一级导航重构 | 当前 `HomePage` 同时承担 `@Entry` 和 Navigation 根容器 | 实现前设计统一根容器，保持现有 AppRouter 参数校验和流程路由，不在页面中复制导航栈 |
| 设置模型迁移 | 当前设置尚无主题色、字号和光感档位 | 新增稳定 Key 和默认值；旧 Preferences 升级后不得读取失败或丢失现有主题 |
| Symbol 名称和效果 | 未逐项确认 | 在 API 26 SDK 中确认候选 Symbol、效果和可用范围 |
| 减少动态能力 | 具体 API 未确认 | 实现前验证；无可靠 API 时提供应用内开关或静态默认 |
| 低端设备档位 | 自动判断方式未确认 | 不猜测设备级别；优先使用可验证能力或明确用户设置 |
| 截图视觉还原 | 参考方向并非 ArkUI 验收证据 | 使用原生 ArkUI 近似实现，不引入第三方 UI/动画库 |
| LIGHT/DARK 对比度 | 建议值尚未测量 | 在最终背景叠加结果上重新测量，不只检查原始色值 |

只有 V1 的设备探针通过后，才能把完整光感写成“已支持”。在此之前，项目文档和答辩表述均应使用“计划采用”“已通过静态编译”或“待设备验证”。

## 11. 范围结论

本方案在不新增业务能力、网络依赖或第三方 UI 库的前提下，将现有一级功能整理为首页、工具、记录、设置四项悬浮 Dock 导航，并增强首页、诊断页和报告页的视觉反馈。记录与设置的业务范围不变，隐私页面仍属于设置页面组；改变的是一级入口组织和全局视觉表现。

底部 Dock 必须保持悬浮发光玻璃形态：完整效果使用 `uiMaterial.ImmersiveMaterial` 和主题色光晕；能力不足时可以降低或关闭材质和动效，但必须保留悬浮位置、圆角轮廓、四项导航和明确选中状态。T17～T24 仍按开发实施计划完成 P0 业务闭环和验收；Dock 与设置扩展应作为独立视觉/导航任务实施，不替代 T20 OCR 验证，也不包含 Form Kit、Natural Language Kit、Data Augmentation Kit 或服务卡片实现。
