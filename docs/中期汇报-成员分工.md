# 开发报错诊断助手中期汇报——成员分工

## 1. 项目当前进度

本项目使用 DevEco Studio、ArkTS 和 ArkUI 开发，采用 HarmonyOS Stage 模型，目标 API Version 为 26。当前已经完成项目需求与架构设计、HarmonyOS 工程基础、主题与公共组件、本地数据访问、首页、本地诊断规则引擎等工作，正在打通“输入错误文本—本地诊断—展示结构化报告”的完整流程。

项目采用分层结构：

```text
页面 Page
  ↓
ViewModel / Store
  ↓
Domain Service
  ↓
Repository
  ↓
Preferences / relationalStore / rawfile
```

页面只负责显示状态和接收操作，不直接执行数据库访问、规则匹配或复杂业务逻辑。

### 2.3 公共数据模型与错误模型

设计并实现了诊断功能使用的公共数据结构，主要位于：

```text
entry/src/main/ets/model/DiagnosisModels.ets
entry/src/main/ets/model/ResultModels.ets
entry/src/main/ets/constants/ErrorCodes.ets
entry/src/main/ets/constants/AppLimits.ets
```

主要模型包括：

- `DiagnosisInput`：保存本次诊断的来源类型和输入文本；
- `DiagnosisReport`：保存错误类型、摘要、证据、可能原因、排查步骤、建议和风险提示；
- `DiagnosisRule`：描述规则关键字、正则表达式、优先级和报告内容；
- `OperationState`：统一表示 `IDLE`、`LOADING`、`SUCCESS`、`ERROR` 和 `CANCELLED`；
- `AppError`：统一错误码、用户提示、是否允许重试及关联编号。

通过统一模型，页面、ViewModel 和 Service 不需要各自定义不同格式的状态和错误对象。

### 2.4 输入校验和诊断会话

主要实现文件：

```text
entry/src/main/ets/domain/validation/InputValidationService.ets
entry/src/main/ets/store/DiagnosisSessionStore.ets
entry/src/main/ets/viewmodel/diagnosis/DiagnosisViewModel.ets
```

实现方式：

1. `InputValidationService` 对输入进行首尾空白规范化，并检查空输入、全空白输入和长度上限；
2. 校验失败时返回统一 `AppError`，不清除用户已经输入的文本；
3. `DiagnosisSessionStore` 保存当前会话的来源类型、原始文本、输入错误和诊断报告；
4. 从报告页返回诊断页时，通过会话 Store 恢复原文，用户可以修改后重新诊断；
5. 页面销毁或异步任务返回时，通过会话标识隔离旧结果，避免旧任务覆盖新输入；
6. 诊断执行期间使用 `OperationState` 防止用户连续点击产生重复任务。

### 2.5 本地诊断规则引擎

主要实现文件：

```text
entry/src/main/ets/domain/diagnosis/RuleMatcher.ets
entry/src/main/ets/domain/diagnosis/DiagnosisService.ets
entry/src/main/resources/rawfile/diagnosis_rules_v1.json
```

实现流程：

```text
用户输入
  ↓
输入规范化与校验
  ↓
加载本地 JSON 规则
  ↓
关键字和正则表达式匹配
  ↓
按 priority 降序、ruleId 升序裁决
  ↓
生成 DiagnosisReport
```

具体实现包括：

- 关键字与正则匹配统一忽略大小写；
- 多条规则同时命中时，先比较规则优先级，再使用 `ruleId` 保证结果稳定；
- 专用规则命中时排除通用规则，避免报告出现互相冲突的分类；
- 没有专用规则命中时，使用 `generic_unclassified` 生成谨慎的通用诊断报告；
- 报告中的证据只描述匹配到的规则信号，不复制 Token、账号、绝对路径等敏感日志内容；
- 损坏正则、非法规则和内部异常统一转换为 `DIAGNOSIS_FAILED`，不向页面暴露底层异常堆栈；
- 规则版本写入诊断报告，便于后续更新规则和定位结果差异。

当前规则已经覆盖 ArkTS 编译错误、Hvigor 构建错误、OHPM 依赖问题、HTTP 错误、超时、权限问题以及未知错误兜底。

## 3. 成员二工作

成员二主要负责平台基础能力、本地数据持久化、页面导航、主题、公共组件和正式首页。

### 3.1 设置存储与主题系统

主要实现文件：

```text
entry/src/main/ets/storage/preferences/PreferencesStore.ets
entry/src/main/ets/repository/settings/SettingsRepository.ets
entry/src/main/ets/viewmodel/settings/SettingsViewModel.ets
entry/src/main/ets/store/AppStateStore.ets
entry/src/main/ets/theme/ThemeModels.ets
```

实现方式：

1. 使用 HarmonyOS Preferences 保存主题模式和基础设置；
2. `SettingsRepository` 负责 Preferences 读写和异常转换，页面不直接访问 Preferences；
3. `SettingsViewModel` 负责设置加载、主题切换、恢复默认和保存状态；
4. `AppStateStore` 作为全局主题单一数据源，保存用户选择主题、系统主题和最终生效主题；
5. SYSTEM 模式根据系统颜色模式解析为 LIGHT 或 DARK；
6. 设置保存失败时，本次会话仍可使用已选择主题，但页面明确提示“仅本次会话生效”；
7. 对延迟读取和连续切换进行了版本隔离，防止较早的异步结果覆盖用户较新的选择。

### 3.2 报告数据库和 Repository

主要实现文件：

```text
entry/src/main/ets/storage/database/DatabaseManager.ets
entry/src/main/ets/storage/database/ReportStore.ets
entry/src/main/ets/storage/database/ReportRowMapper.ets
entry/src/main/ets/repository/report/LocalReportRepository.ets
```

实现方式：

- 使用 HarmonyOS relationalStore 保存用户主动保存的诊断报告；
- `DatabaseManager` 负责数据库初始化和版本管理；
- `ReportRowMapper` 负责 `DiagnosisReport` 与数据库行之间的转换；
- `ReportStore` 负责底层增、查、删和事务操作；
- `LocalReportRepository` 为 ViewModel 提供保存报告、读取最近记录、读取详情和删除记录等统一接口；
- 数据库异常转换为应用错误，页面不接触 SQL、数据库字段或平台异常；
- 通过报告 ID 防止同一报告被重复保存；
- 最近记录按保存时间倒序读取，首页最多取三条。

### 3.3 页面路由

主要实现文件：

```text
entry/src/main/ets/router/AppRouter.ets
entry/src/main/ets/router/RouteModels.ets
entry/src/main/ets/router/RouteNames.ets
entry/src/main/ets/router/RouteRegistry.ets
```

项目使用 ArkUI `Navigation` 和 `NavPathStack` 管理页面，统一注册以下路由：

- `home`；
- `diagnosis`；
- `report`；
- `tools`；
- `historySettings`；
- `privacy`。

实现了路由参数白名单校验。例如，诊断页只接受 `IMAGE` 或 `TEXT` 来源；报告页只接受 `sessionId` 或 `reportId`；禁止把完整日志、图片数据和完整报告对象放入路由参数。未知路由会安全返回首页，重复打开同一顶层页面不会不断压栈。

### 3.4 公共状态组件

主要实现文件：

```text
entry/src/main/ets/components/common/AsyncStatePanel.ets
entry/src/main/ets/components/common/EmptyStateView.ets
entry/src/main/ets/components/common/InlineErrorNotice.ets
entry/src/main/ets/components/common/ConfirmActionDialog.ets
```

实现内容：

- `AsyncStatePanel`：统一加载、成功、失败、取消和重试状态；
- `EmptyStateView`：统一无历史、无结果等空数据状态；
- `InlineErrorNotice`：在当前内容附近展示错误原因和下一步操作；
- `ConfirmActionDialog`：用于删除和清空等危险操作的二次确认。

组件只接收状态和回调，不访问 Repository、数据库或网络。所有关键按钮最小高度为 44vp，错误状态同时使用标题和文字，不只依靠红色表达。

### 3.5 首页与最近记录

主要实现文件：

```text
entry/src/main/ets/pages/home/HomePage.ets
entry/src/main/ets/viewmodel/home/HomeViewModel.ets
entry/src/main/ets/viewmodel/home/HomeNavigation.ets
entry/src/main/ets/components/history/HistoryReportCard.ets
```

实现方式：

1. 首页同步显示截图诊断、文本诊断和验证工具入口，不等待数据库加载；
2. `HomeViewModel` 异步调用 `LocalReportRepository.getRecent(3)` 读取最近三条报告；
3. 数据库加载失败只影响最近记录区域，不阻塞诊断入口；
4. 最近记录存在时，使用 `HistoryReportCard` 展示错误类型、脱敏摘要、保存时间和来源；
5. 无记录时复用 `EmptyStateView`，读取失败时复用 `InlineErrorNotice` 并提供重试；
6. 点击入口和历史记录均通过 `HomeNavigation` 与 `AppRouter` 跳转，页面不直接拼装复杂路由；
7. 首页使用 Scroll 布局，在 1.5 倍字体下仍能滚动访问全部操作。
