# “移动软件开发”课设报告撰写要求

> 适用学期：2025—2026 学年第二学期  
> 课程名称：移动软件开发  
> 整理用途：指导本项目后续撰写、检查和提交课设报告  
> 整理依据：`docs/requirements/report.md`、`docs/requirements/考核要求带图片.md`、`docs/requirements/移动软件开发要求.md`，以及课程提供的 Word 报告模板  
> 说明：本文只提取和整理现有材料中的要求。标为“建议”的内容是为了便于执行，不等同于老师新增的硬性规定。

## 1. 核心硬性要求

1. 必须使用课程提供的 `2. “移动软件开发”课设报告-组别-姓名-姓名（模板）.docx` 撰写最终报告。
2. 报告中的标题、正文、图、表等内容必须遵循模板规定的格式，不应自行更换整体版式。
3. 报告正文总字数控制在 **3000 字以内**。
4. 最终文件应使用 DOCX 格式提交，文件名为：

   `“移动软件开发”课设报告-组别-姓名-姓名.docx`

   示例：

   `“移动软件开发”课设报告-TEAM00-张三-李四.docx`

5. 报告放在最终考核文件夹根目录，不放入 `源代码` 文件夹。
6. 报告内容必须与最终能够运行和演示的程序一致，不能把计划功能、未完成能力或未经验证的结果写成已经实现。
7. 报告和 PPT 可以使用大模型辅助生成，但必须人工核对和修改，项目成员需要能够解释报告涉及的功能、代码、数据流和关键 API。

## 2. 报告封面要求

封面沿用模板，不自行改变学校名称、学期、课程名称和指导教师等固定信息。需要填写的内容如下：

| 字段 | 模板内容或填写要求 |
| --- | --- |
| 学校 | 北京工业大学 |
| 学期 | 2025—2026 学年第二学期 |
| 文档类型 | 课设报告 |
| 课程名称 | 移动软件开发 |
| 专业 | 软件工程（实验班），如实际专业不同则按真实信息填写 |
| 组别 | 填写实际 TEAM 编号 |
| 项目组组长 | 填写组长学号、姓名 |
| 项目组成员 | 逐人填写学号、姓名；人数应与实际分组一致 |
| 指导教师 | 贺国平、王岚 |

注意事项：

- 每组最多 5 人。
- 每名列入报告的成员都必须实际参与编码，不能仅负责报告或 PPT。
- 成员姓名、学号、组别必须与答辩 PPT、提交目录和压缩包命名一致。
- 如果当前项目实际为个人开发，应只保留真实参与者信息，不虚构其他成员。

## 3. 目录要求与模板矛盾

报告应包含自动目录，并在最终定稿后更新目录页码。模板材料存在以下不一致：

- 目录示例列为：绪论、需求分析、系统设计、系统实现、系统测试、总结，共 6 章；
- 正文示例列为：绪论、需求分析、系统设计与实现、系统测试、总结，共 5 章。

执行建议：

1. 最终排版时优先保留课程 Word 模板中的样式和标题层级。
2. 如果正式模板正文仍将“系统设计与实现”合并，则正文采用 5 章结构，并重新生成目录，保证目录与正文完全一致。
3. 如果老师要求严格使用 6 章结构，则将第 3 章写为“系统设计”、第 4 章写为“系统实现”，系统测试和总结顺延。
4. 在未向老师确认前，不应出现目录有“系统实现”而正文没有对应章节的情况。

本文后续以模板正文展示的 **5 章结构**整理写作要求；拆分为 6 章时只需把“设计”和“实现”分开，内容要求不变。

## 4. 正文章节及内容要求

### 4.1 第 1 章 绪论

#### 4.1.1 系统的主要内容

本节用于说明项目为什么做、为谁解决什么问题，以及最终形成了什么应用。建议至少包含：

- 项目背景与问题来源；
- 目标用户和典型使用场景；
- 系统的核心目标；
- 主要功能概览；
- 应用作为一个完整产品的主题与价值；
- 第一版实际完成范围，以及没有完成的增强功能边界。

对于本项目，应围绕“鸿蒙开发错误诊断”主题展开，说明应用如何接收 ArkTS、Hvigor、OHPM 等错误信息，通过规则或 AI 辅助分析生成诊断报告，并提供本地保存、历史记录、收藏和设置等支撑功能。

不要在本节堆叠实现细节，也不要把多个互不相关的演示页面描述成一个完整系统。

#### 4.1.2 系统涉及的技术知识点

本节用于说明项目采用的鸿蒙技术与工程技术。应以实际代码和实际验证结果为准，可包含：

- 最新版 DevEco Studio 的实际版本；
- HarmonyOS/OpenHarmony SDK 版本和 API Version，必须不低于 26；
- ArkTS 与 ArkUI；
- Stage 模型；
- 页面路由和状态管理；
- 本地数据存储；
- 网络请求和 OpenAI 兼容接口；
- 日志与异常处理；
- 单元测试、页面测试和 Mock；
- 项目实际使用的鸿蒙新能力，例如 Form Kit、Core Vision OCR、沉浸式或光感效果等；
- 模拟器、真机或远程真机的验证情况。

技术知识点不能只列名称。建议用简短文字说明“技术用途、使用位置、解决的问题和实际效果”。对于尚未完成真机验证的能力，应明确写为“静态检查通过”或“待真机验证”，不能写成已经在设备上成功运行。

### 4.2 第 2 章 系统的需求分析

本章应说明系统要实现什么，重点来自需求规格说明书，而不是重复代码结构。建议包含：

- 用户角色或主要使用者；
- 典型使用场景；
- 核心业务流程；
- 功能需求及优先级；
- 非功能需求；
- 运行环境与约束；
- 隐私、安全和权限要求；
- 第一版范围。

需求描述应与项目文档统一：

- P0 是首个稳定基线；
- P1 是课程第一版的增强交付；
- Data Augmentation Kit 统一为 P1；
- 隐私页面归入“记录与设置”页面组；
- 服务卡片是系统桌面入口，不作为独立页面组。

建议使用用例图、业务流程图或功能结构图辅助说明。所有图必须对应真实需求，并在正文中通过“如图 2.x 所示”进行引用。

### 4.3 第 3 章 系统设计与实现

本章是报告主体，应把总体设计、关键模块和实际界面结合起来，说明系统如何落地。可根据项目规模设置二级、三级标题，但标题应具体，不能保留模板中的“***”占位符。

建议至少包含以下内容。

#### 4.3.1 总体架构设计

- 系统分层结构；
- 页面展示层、ViewModel/业务逻辑层、Repository、Service、本地存储和公共基础能力之间的关系；
- 主要依赖方向；
- 数据从用户输入到诊断结果、再到本地历史记录的流向。

建议插入总体架构图。图中名称应与需求规格说明书、概要设计说明书、详细设计说明书及代码目录保持一致。

#### 4.3.2 功能模块设计

说明实际实现的主要模块，例如：

- 首页与导航；
- 文本输入与诊断；
- 规则诊断与报告生成；
- 诊断报告展示；
- 历史记录、收藏和删除；
- 记录与设置；
- AI 与网络配置；
- OCR、服务卡片或其他鸿蒙新能力；
- 日志、异常和公共组件。

每个模块建议说明：职责、入口、输入、输出、依赖和实现状态。

#### 4.3.3 界面与导航设计

- 展示主要页面截图；
- 说明页面入口、主要操作和页面跳转；
- 说明加载、空数据、错误、权限不足等状态；
- 说明手机端布局和主题适配；
- 说明发光玻璃、莫兰迪配色等视觉方案与可读性处理。

页面截图必须来自最终可运行版本，截图中的文字、按钮和状态应清晰，不应保留调试遮挡、敏感信息或明显未完成内容。

#### 4.3.4 核心业务流程实现

建议围绕一条完整主流程展开：

1. 用户输入或导入错误文本；
2. 页面校验输入；
3. ViewModel 调用诊断服务；
4. 规则或 AI 服务分析错误；
5. 系统生成诊断报告；
6. 用户查看原因、步骤和风险提示；
7. 用户保存、收藏、取消收藏或删除历史报告。

建议使用时序图或流程图，并结合关键界面截图说明。报告不需要粘贴大段完整源码，可选择少量关键代码片段解释设计要点。

#### 4.3.5 数据与本地存储实现

- 主要数据模型；
- 诊断报告、历史记录、收藏状态和设置数据之间的关系；
- 哪些数据只在内存保存，哪些数据持久化；
- 数据更新、删除和失效策略；
- Token、源代码和隐私数据的保护方式。

#### 4.3.6 鸿蒙新能力应用

课程要求尽可能使用鸿蒙新能力。本节应只写实际采用或实际验证过的能力，并说明：

- 能力名称；
- 使用目的；
- 对应模块或页面；
- SDK、权限、设备和签名条件；
- 模拟器或真机验证结果；
- 不可用时的降级方案；
- 对项目体验或技术价值的提升。

老师没有规定新能力最低数量。报告应优先突出与项目主题相关、稳定且可演示的能力，不能为了数量虚构集成结果。

### 4.4 第 4 章 系统测试

本章应提供能够支撑“项目可编译、可运行、核心流程可重复演示”的证据。建议包含：

- 测试环境：操作系统、DevEco Studio、SDK/API、模拟器或设备；
- 构建测试：OHPM、Hvigor、ArkTS 编译、Debug HAP；
- 单元测试：主要 Service、Repository、ViewModel、数据转换和公共模型；
- 页面或流程测试：输入、诊断、报告、保存、历史、收藏、删除和设置；
- 异常测试：空输入、无网络、请求超时、接口不可用、存储失败、权限拒绝；
- 兼容性测试：浅色/深色模式、手机/平板或不同窗口尺寸；
- 鸿蒙新能力测试：模拟器和真机分别记录，未验证项如实标注；
- 测试结论及已知问题。

测试结果适合用简洁表格表示，至少包含：编号、测试对象、操作或输入、预期结果、实际结果、结论。应优先放入关键流程和异常场景，不要只写“测试通过”而没有条件和结果。

如果使用测试截图，截图应能证明对应测试确实执行，例如构建成功日志、测试报告、模拟器运行页面或权限提示。不得在截图中暴露 Token、账号、证书、私钥或密码。

### 4.5 第 5 章 总结

总结应包含项目总结和成员个人总结。

#### 4.5.1 项目总体总结

- 项目解决了什么问题；
- 完成了哪些核心功能；
- 使用了哪些关键鸿蒙技术和新能力；
- 项目当前达到的稳定程度；
- 开发过程中遇到的主要问题及解决办法；
- 当前限制和后续改进方向。

总结必须与实际完成度一致。未完成的 OCR 真机证据、完整自动化测试或 P1 能力，应作为限制或后续工作，不写成已完成成果。

#### 4.5.2 成员个人总结

模板为每位成员预留“姓名、学号”小节。每名成员应分别说明：

- 自己负责的功能；
- 对应模块和主要代码位置；
- 使用的 ArkTS、ArkUI 或鸿蒙 API；
- 完成的具体工作和可演示结果；
- 遇到的一个关键问题及解决过程；
- 个人收获与不足。

个人总结必须与 Git 提交、代码目录、成员分工和现场答辩内容相互对应。不能出现成员只负责文档而没有编码成果的描述。

## 5. 图、流程图、UML 图和截图要求

### 5.1 插图编号与图题

模板明确要求：

1. 插图逐章单独编号，例如“图 1.1”“图 2.3”“图 6.8”。
2. 图序必须连续，不得重复或跳号。
3. 图与表采用一致的分章编号规则。
4. 全文只有一幅图时，在图题前使用“附图”字样。
5. 一幅图包含多个分图时，使用 a、b、c……标识分图。
6. 图序和图题置于图片下方居中位置。
7. 正文必须引用对应图片，例如“系统总体架构如图 3.1 所示”。

### 5.2 流程图和 UML 图

工作阶段可以使用 Mermaid 编写架构图、流程图、状态图、类图和时序图。转入最终 Word 报告时，应将 Mermaid 图导出为清晰图片，并按照模板的插图规则插入和编号。

建议使用：

- `flowchart`：系统架构、页面导航、业务流程；
- `sequenceDiagram`：诊断、保存、收藏等调用链；
- `classDiagram`：主要数据模型或模块依赖；
- `stateDiagram-v2`：页面或诊断任务状态变化。

图中应使用统一术语，避免出现代码名称、文档名称和界面名称三套不同叫法。

### 5.3 页面截图与运行证据

课程材料没有规定必须放多少张截图，但为了证明完整应用和个人实际成果，建议至少保留：

- 首页或主要入口；
- 错误输入或导入页面；
- 诊断结果页面；
- 历史记录及收藏状态；
- 设置或隐私页面；
- 实际采用的鸿蒙新能力效果；
- 构建成功或测试执行证据。

截图要求：

- 来自最终或接近最终的可运行版本；
- 内容完整、文字清晰、裁剪合理；
- 不出现真实 Token、账号、签名信息、隐私图片或本机敏感路径；
- 不使用无法复现的效果图冒充运行截图；
- 图题说明页面或测试所证明的内容；
- 同类截图数量适度，避免挤占 3000 字以内报告的有效篇幅。

## 6. 表格要求

模板明确要求：

1. 表格逐章单独编号，例如“表 1.1”“表 2.5”。
2. 表序必须连续，不得重复或跳号。
3. 表序与表题放在表格上方居中位置。
4. 没有表题时，表序放在表格左上方或右上方；全文位置应保持一致。
5. 各栏应标注量及相应单位。
6. 表内数字上下对齐。
7. 相邻栏数值相同时仍需逐项填写，不能写“同上”“同左”等替代词。
8. 表格结构应简洁，避免放入大段难以阅读的正文。

建议优先使用表格表达：功能与优先级、模块职责、测试用例、成员分工、开发环境和完成状态。

## 7. 内容真实性与技术口径

报告中的开发环境和项目结论应遵循以下口径：

- 使用最新版 DevEco Studio；
- SDK/API Version 不低于 26；
- 使用 ArkTS 开发；
- 项目是鸿蒙手机端应用，不是 Android、iOS 或普通 Web 项目；
- 后端和 SQL 文件为可选内容，不应为了报告虚构后端；
- 模拟器可以作为主要开发和演示环境；
- 必须真机验证的能力应分别注明“真机已验证”或“待真机验证”；
- 远程真机和鸿蒙新能力属于加分方向，但现有材料没有给出具体加分分值；
- AI 辅助开发可以写入开发过程，但成员必须理解并能解释最终代码；
- 所有功能、截图、测试结果、成员贡献和已知问题必须与最终仓库一致。

## 8. 字数控制

正式要求为正文总字数不超过 3000 字。现有材料没有明确说明封面、目录、图题、表题、表格文字和个人总结是否计入 3000 字，因此最终应按偏保守口径控制。

建议正式报告字数分配如下：

| 章节 | 建议字数 |
| --- | ---: |
| 绪论 | 350—450 字 |
| 需求分析 | 400—500 字 |
| 系统设计与实现 | 1100—1400 字 |
| 系统测试 | 400—550 字 |
| 总结及个人总结 | 350—500 字 |
| 合计 | 约 2600—3000 字 |

图表用于压缩和结构化信息，但不能用大量截图替代必要说明。工作初稿可以先写详细，再在最终排版阶段压缩到 3000 字以内。

## 9. 报告与评分、答辩的关系

实验/课设报告占课程考核的 30%。完整考核构成为：

| 考核项 | 比例 |
| --- | ---: |
| 实验过程 | 30% |
| 演示答辩 | 20% |
| 实验/课设报告 | 30% |
| 平时考核 | 20% |

因此报告应能够辅助答辩证明以下事项：

- 项目具有统一主题和完整核心流程；
- 应用能够编译、安装、启动和运行；
- 项目实际采用 ArkTS、API 26 以上版本和鸿蒙能力；
- 每位成员都有明确代码贡献；
- 关键功能有运行和测试证据；
- 开发者理解系统架构、数据流、关键 API、问题和解决方案。

报告不应只描述界面外观，也不应只粘贴代码。应通过“需求—设计—实现—测试—总结”的证据链说明项目确实完成。
