# CodePulse 现场演示流程与提问应对

> 适用场景：老师不要求完整讲述 PPT，而是要求学生一边操作 App，一边说明设计与实现。
>
> 建议时长：6～7 分钟。若现场时间不足，可使用文末的 3 分钟压缩流程。

## 一、演示前准备

答辩开始前完成以下检查，不占正式演示时间：

- 启动模拟器并打开 App，停留在首页。
- 准备一段固定的脱敏错误日志，并复制到剪贴板。
- 历史记录中预留 1～2 条可删除的测试记录。
- 使用本地规则诊断或本地 Skill，不在现场输入真实 API Key。
- 保持网络能力、OCR、服务卡片等未稳定验证功能关闭或不进入。
- 关闭无关窗口和通知，避免屏幕出现账号、路径、Token 等隐私信息。
- 提前确认“诊断、保存、收藏、打开历史记录、删除记录”主流程能够正常运行。

推荐演示日志：

```text
hvigor ERROR: task failed
> Task :entry:CompileArkTS
ArkTS:ERROR Example.ets:12:5
Type 'string' is not assignable to type 'number'.
BUILD FAILED
```

这段日志同时包含 ArkTS 类型错误和 Hvigor 构建失败包装信息，适合说明多规则命中与主错误裁决。

---

## 二、6～7 分钟标准演示流程

### 第 1 步：首页与项目定位（约 30 秒）

**现场操作**

打开首页，短暂停留，依次指向“文本诊断”“最近诊断记录”和底部导航。

**建议讲解**

> 这是一个使用 ArkTS、ArkUI 和 Stage 模型开发的 HarmonyOS 错误诊断工具。它主要面向鸿蒙开发过程中较长、较杂的 ArkTS、Hvigor 和 OHPM 错误日志，把原始日志转换成结构化的错误类型、证据、原因、修复步骤和风险提示。项目采用本地优先设计，即使没有网络也能完成核心诊断。
>
> 我在需求分析时把用户的痛点归纳成三点：第一，真正的根因容易被最后一行 `BUILD FAILED` 之类的包装信息掩盖；第二，开发者知道“构建失败”，但不一定知道下一步应该修改什么；第三，错误处理完成后缺少统一记录，遇到相似问题时还要重新搜索。因此首页把文本诊断作为主要入口，同时提供最近记录和历史回看，让一次诊断形成完整闭环。
>
> 工程上没有把所有逻辑写在页面里，而是按页面、ViewModel、Service、Repository 和 Storage 分层。界面变化不会直接影响规则和存储，增加诊断规则时也不需要重新设计页面。

**要讲的重点**

- 项目不是替代 DevEco Studio，而是补充其错误整理和诊断能力。
- P0 核心闭环是“输入日志—诊断—生成报告—保存—历史管理”。
- 页面只负责展示和事件转发，业务判断、数据访问和存储分层处理。

**老师可能提问**

1. **DevEco Studio 已经会显示错误，为什么还要做这个 App？**
   - DevEco 能定位编译错误，但长日志中经常混有上层构建失败、底层类型错误和重复堆栈。本项目的价值是提取核心错误、给出证据和可执行修复步骤，并保存成可回看的诊断报告。

2. **为什么不直接全部交给大模型？**
   - 本地规则响应稳定、无需网络、结果可解释，也不会默认上传代码。大模型是增强能力，不作为 P0 核心链路的唯一依赖。

**故障备用**

如果首页最近记录加载异常，先演示文本诊断；历史记录可在后续通过预留截图或代码结构说明，不在现场反复重启应用。

### 第 2 步：输入校验与日志提交（约 45 秒）

**现场操作**

1. 点击“文本诊断”。
2. 可先在空输入状态点击一次诊断，展示输入校验提示。
3. 粘贴准备好的错误日志。
4. 点击开始诊断。

**建议讲解**

> 输入页不会直接发网络请求或写数据库，而是把事件交给 ViewModel。系统会先执行去除空白、空内容校验和长度限制，再进入诊断服务。这样可以避免无效请求，也能防止超长日志造成内存和正则匹配压力。
>
> 这里可以看到，空输入不会进入诊断流程，而是直接给出明确提示；粘贴日志后，原始文本仍保留在输入框中，用户可以继续修改。点击诊断后，页面进入加载状态，ViewModel 负责阻止同一页面重复提交，并把成功结果或统一错误状态返回给页面。
>
> 输入校验虽然简单，但它是整个流程的第一道稳定性保护。错误日志可能来自终端、DevEco Studio 或第三方工具，格式并不统一，所以系统只做必要的归一化，不会擅自删除可能影响判断的关键内容。

**要讲的重点**

- 校验失败时保留原输入，用户可直接修改。
- 设置最大输入长度是资源保护边界，不是宣称已经完成所有极限性能测试。
- 诊断期间通过加载状态阻止同一页面重复提交。

**老师可能提问**

1. **为什么限制约 10 万字符？**
   - 这是第一版基于移动端内存、正则扫描成本和可用性的工程边界。后续若处理大型日志，应改成分块、流式解析，并用基准测试重新确定限制。

2. **为什么页面不直接调用诊断方法？**
   - 页面只处理输入和展示，ViewModel 组织状态，Service 承担校验和诊断流程。这样便于单元测试，也能避免页面同时承担网络、存储和业务逻辑。

**故障备用**

如果剪贴板粘贴失败，手动输入一条简短错误：

```text
ArkTS:ERROR Type 'string' is not assignable to type 'number'.
```

### 第 3 步：错误类型匹配与裁决（约 60 秒）

**现场操作**

等待诊断完成并进入诊断报告页，指向识别出的错误类型和诊断证据。

**建议讲解**

> 系统会让每条规则通过关键字或正则表达式匹配日志，收集全部命中的候选规则，再按照优先级从高到低、规则编号从小到大进行确定性裁决。这里不是优先队列，也不是依赖运行时偶然顺序的所谓稳定排序。以当前日志为例，ArkTS 类型错误和 Hvigor 构建失败都会命中，但 ArkTS 规则优先级更高，因此被选为主错误，Hvigor 作为上层构建失败信息保留。
>
> 一条规则除了匹配条件，还包含错误类别、匹配证据、可能原因、修复建议、风险提示和优先级。关键字适合识别稳定的错误短语，正则表达式适合识别文件位置、错误码或带有变化部分的文本。命中后不会立即返回，而是继续收集候选，避免日志前面的普通警告覆盖后面更具体的根因。
>
> 这段示例中，`Type 'string' is not assignable to type 'number'` 是可以直接指导代码修改的具体类型错误，而 `BUILD FAILED` 只是构建任务最终失败的结果。因此系统优先展示 ArkTS 类型错误，并保留 Hvigor 信息作为辅助证据。若没有已知规则命中，系统仍会生成通用报告，而不是显示空白页。

**要讲的重点**

- 匹配条件：关键字或正则表达式命中。
- 多规则并非只取第一个，而是先收集候选，再进行确定性优先级裁决。
- 规则结果包含证据、原因、修复步骤和风险，而不只是错误标签。
- 未命中已知规则时返回通用诊断结果，避免页面无响应。

**老师可能提问**

1. **错误类型具体是怎么匹配的？**
   - 每条规则定义适用来源、关键字或正则、优先级和诊断模板。日志归一化后逐条匹配，收集候选结果，再按优先级和规则编号确定主结果。

2. **会不会误判？**
   - 会存在边界情况，例如短关键字可能出现在普通文本中。当前通过正则、组合特征和优先级降低误判；后续可增加词边界、负向条件、置信度和真实日志回归集。

3. **规则多了性能会不会下降？**
   - 当前规则数量较少，线性扫描足够。粗略成本与规则数、日志长度和匹配表达式有关；规模扩大后可按工具来源预分类、建立关键字索引，并限制高成本正则。

4. **为什么 ArkTS 是主错误而 Hvigor 不是？**
   - `BUILD FAILED` 往往只是上层结果，真正能指导修复的是具体的类型错误。因此细粒度根因规则优先级高于包装性构建失败规则。

**故障备用**

若预期规则没有命中，应如实说明当前规则集存在覆盖边界，展示通用诊断结果，并用准备好的报告记录继续后续演示，不要现场修改代码。

### 第 4 步：结构化诊断报告（约 45 秒）

**现场操作**

在报告页依次展示错误类型、诊断证据、可能原因、修复建议、风险提示和“展开原文”。

**建议讲解**

> 报告页不是只展示一个错误名称，而是把诊断结果拆成证据、原因、处理步骤和风险提示。用户可以核对规则为什么命中，也可以展开原始日志。一次诊断通过会话标识和报告标识串联，保存后可从历史记录重新打开同一份报告。
>
> 其中“证据”回答系统为什么作出这个判断，“可能原因”帮助用户理解错误产生的机制，“修复步骤”给出可以按顺序执行的操作，“风险提示”避免用户为了消除报错而关闭类型检查或隐藏真正问题。报告的目标不是替用户盲目修改代码，而是把原始日志整理成可以核对、可以执行的诊断建议。
>
> 报告页保留诊断原文的展开入口，用户可以随时对照上下文，避免摘要造成信息丢失。用户选择修改文本并重新诊断时，会进入新的诊断过程，而不是直接篡改已经保存的历史记录。

**老师可能提问**

1. **如何保证结果可解释？**
   - 每条规则保存匹配证据和固定的诊断内容，报告会展示触发判断的日志特征，而不是只给出无法追溯的结论。

2. **原始日志保存在哪里？**
   - 未保存的输入主要存在当前运行内存中；用户主动保存报告后，报告及原文会进入应用本地数据库。因此原文属于敏感数据，必须允许用户删除，也禁止写入普通日志。

**故障备用**

若报告页滚动位置不便展示，优先说明“错误类型—证据—修复步骤”三个核心部分，不必逐字朗读全部内容。

### 第 5 步：保存、收藏与历史回看（约 75 秒）

**现场操作**

1. 点击“保存报告”，展示按钮变为“已保存”。
2. 点击“收藏报告”，确认按钮立即变为“取消收藏”。
3. 进入底部“记录”页面。
4. 找到刚保存的报告并重新打开。
5. 返回后可再切换一次收藏状态。

**建议讲解**

> 报告使用关系型本地数据库保存，便于排序、去重、收藏和删除。保存通过报告 ID 避免同一份报告重复插入。收藏按钮采用即时状态更新，让界面立即反馈；如果持久化失败，则回滚界面状态并提示用户。历史记录默认按时间展示，并设置最多 200 条的容量边界。
>
> 这里点击保存后，按钮显示“已保存”，说明保存操作已经完成，而不只是页面内改变文字。进入记录页后，可以通过报告 ID 找到刚才的同一条数据，并重新进入报告详情。收藏也是持久化字段，因此退出详情页再返回时，收藏状态仍然存在。
>
> 我把历史记录和普通设置分开存储：报告具有多个字段，并且需要查询、更新、排序和删除，因此使用关系型数据库；主题、字号和视觉效果只是少量键值配置，更适合 Preferences。这个选择符合两类数据各自的特点，也避免为简单配置设计不必要的数据表。

**要讲的重点**

- 保存是用户主动操作，不是默认收集全部输入。
- 收藏状态必须立即反馈，并与数据库状态保持一致。
- 历史记录和设置数据采用不同存储方式：结构化记录使用关系型数据库，轻量设置使用 Preferences。

**老师可能提问**

1. **为什么历史记录不用 Preferences？**
   - 历史记录需要按时间查询、按 ID 更新、收藏和删除，关系型数据库更适合结构化查询；Preferences 更适合主题、字号等少量键值配置。

2. **如果已经有 200 条并且全部收藏，会怎么样？**
   - 系统不会为了腾出空间删除收藏记录，会返回“历史记录受保护”的错误。第一版选择保护用户收藏，后续可让用户调整容量或手动清理。

3. **连续快速点击保存或收藏会不会重复？**
   - 页面有加载保护，保存还通过报告 ID 去重。收藏采用当前状态更新并在失败时回滚；跨页面并发仍可继续通过数据库事务和版本字段增强。

4. **数据库写入和超量清理是不是一个事务？**
   - 当前插入和清理不是完整单事务，这是已识别的技术债。生产版本应将容量检查、插入和清理合并进事务，防止进程中断导致短暂超量。

**故障备用**

如果刚保存的记录未立即出现，返回首页或记录页触发一次刷新。不要重复多次点击保存，以免干扰演示数据。

### 第 6 步：删除与清空保护（约 40 秒）

**现场操作**

1. 对预留的测试记录点击“删除”。
2. 先展示二次确认并点击取消。
3. 再次删除并确认。
4. 点击“清空全部历史记录”，只展示确认框后取消，不必真的清空。

**建议讲解**

> 删除单条记录和清空全部历史都需要二次确认，避免误操作。系统先完成数据库操作，再从页面列表移除数据；如果数据库失败，页面不会显示虚假的删除成功。
>
> 单条删除和清空全部是两个不同风险级别的操作，但都不能只靠按钮颜色提醒，所以统一增加确认对话框。演示时先取消一次，可以说明用户仍然拥有最终决定权；再次确认后，Repository 执行实际删除，ViewModel 再刷新页面状态。
>
> 这个处理顺序强调数据库才是真实数据来源。如果先把列表项从界面移除、数据库随后写入失败，用户会误以为记录已经删除；现在只有持久化成功后才更新界面，从而减少界面与数据库状态不一致的问题。

**老师可能提问**

1. **为什么一定要二次确认？**
   - 删除和清空属于不可逆操作，尤其报告中可能包含用户整理过的诊断结果，因此需要明确确认。

2. **应用在保存过程中被杀死会怎样？**
   - 单次数据库写入具有原子性，但插入与容量清理目前不是同一事务。核心记录不会以半条形式出现，但容量状态可能需要下次启动时再次校正。

**故障备用**

若不想改变现场数据，展示确认框后统一点击取消，也足以证明防误触流程存在。

### 第 7 步：本地 Skill 与可选 AI（约 50 秒）

**现场操作**

进入“智能”页面，切换到 Skill，选择一个稳定工具，例如“错误分诊”或“JSON 格式化”，输入固定示例并执行。

**建议讲解**

> 智能模块把能力拆成单一职责的 Skill，例如错误分诊、JSON 格式化、报告整理和隐私脱敏。现场演示的是本地能力，不需要网络。项目也预留 OpenAI 兼容接口，但只有用户主动配置后才会调用；API Key 只保存在当前运行内存，不写入本地数据库。
>
> Skill 可以理解为针对明确任务封装的小工具：错误分诊负责判断日志的大致来源，JSON 格式化负责检查和整理结构化数据，报告整理负责组织零散信息，隐私脱敏用于处理可能出现的敏感字段。每个 Skill 的输入和输出边界清楚，便于单独测试，也避免把所有能力塞进一个无法解释的“万能 AI”按钮。
>
> 在线模型的作用是补充规则结果，例如解释复杂上下文或把建议组织得更自然，但它不会根据输出自动执行命令，也不是核心流程的强制依赖。这样设计主要考虑网络波动、调用费用、API Key 安全和源代码上传风险。答辩现场使用本地能力，是为了展示稳定可复现的实现，而不是回避在线接口。

**要讲的重点**

- 本地规则和 Skill 是稳定的演示路径。
- 在线模型属于可选增强，不应包装成已经完全本地实现的大模型。
- 用户提供 API Key 后，如果在线请求失败，当前实现会显示错误；用户可手动切回本地模式，并非所有情况都会自动降级。

**老师可能提问**

1. **你这里真的用了大模型吗？**
   - 项目提供 OpenAI 兼容的在线模型接入，但 P0 核心诊断不依赖大模型。没有配置 Key 时使用本地规则和本地 Skill，保证离线可用和结果可解释。

2. **后端是 AI 服务吗？**
   - 当前演示后端主要用于接口链路和诊断服务验证，核心仍是规则匹配。真正的生成式模型通过兼容接口接入，不能把简单关键字接口描述成训练后的 AI 模型。

3. **在线 AI 失败会自动切本地吗？**
   - 未配置 Key或本地后端不可达时可以使用本地能力；已配置在线 Key 后请求失败，当前会明确展示错误，需要用户手动切回本地模式。

4. **如何防止日志中的提示词注入？**
   - 当前不会根据模型输出自动执行命令，降低了直接风险；但提示词注入、输入分隔和输出校验仍属于后续需要加强的安全项。

**故障备用**

不在答辩现场输入 Key 或调用不稳定网络。若 Skill 页面异常，可回到已生成的诊断报告，说明智能能力是增强模块，不影响 P0 本地诊断闭环。

### 第 8 步：设置、隐私与结束总结（约 40 秒）

**现场操作**

进入“设置”页面，展示主题、字号或视觉效果设置，再打开隐私说明页面。

**建议讲解**

> 设置使用轻量键值存储，并通过全局状态同步到页面。项目采用统一语义色，支持明暗主题和基本视觉效果。隐私方面，未保存日志不主动持久化，敏感原文和 API Key禁止写入普通日志，用户可以删除单条记录或清空全部历史。
>
> 主题、字号和视觉效果属于跨页面共享状态，所以用户在这里修改后，其他页面能够同步显示，而不是每个页面各自维护一份配置。界面配色采用灰蓝、鼠尾草绿和陶土红：灰蓝用于主要操作，鼠尾草绿用于收藏或成功状态，陶土红用于删除和风险操作。同时通过文字和确认框表达状态，避免只依赖颜色区分。
>
> 隐私方面，应用不会在用户没有点击保存时主动把整段日志写入历史数据库；用户主动保存后，原文属于本地报告的一部分，因此提供单条删除和清空全部历史。普通运行日志只记录错误码、操作类型或追踪信息，不应记录完整源代码、Token 和 API Key。
>
> 整个演示展示了从日志输入、确定性诊断、结构化报告，到本地保存、收藏和历史管理的完整闭环。这个闭环是当前可以稳定运行和复现的 P0 基线。OCR、服务卡片及部分 Kit 能力仍需真机或特定环境验证，因此本次没有把静态检查或未闭环能力描述成已经运行成功。

**老师可能提问**

1. **本地数据库里的日志加密了吗？**
   - 当前依赖 HarmonyOS 应用沙箱隔离，没有实现字段级加密。若进入生产环境，应增加敏感字段加密、可选不保存原文和更细的数据保留策略。

2. **为什么不演示 OCR 或服务卡片？**
   - 这些能力受 SDK、设备、签名或桌面环境影响，目前缺少稳定的真机闭环证据。答辩只展示已经实际完成和可复现的功能，避免把静态验证说成运行成功。

3. **项目目前最大的不足是什么？**
   - 规则覆盖依赖真实错误样本积累；部分边界匹配仍可能误判；数据库容量处理还可进一步事务化；OCR 和部分系统 Kit 仍需真机验证。

## 四、老师打断时的应答方式

### 1. 老师问实现细节

先停止操作，用“结论—证据—不足”三句话回答：

> 结论是系统采用关键字和正则匹配，并进行确定性优先级裁决。证据是这段日志同时命中 ArkTS 和 Hvigor 规则，最终选择更具体的 ArkTS 类型错误。当前不足是规则覆盖和边界条件仍需更多真实样本验证。

### 2. 老师指出功能没有完成

不要辩解或把静态检查说成运行成功：

> 这个能力目前只完成了静态兼容性检查或部分代码验证，还没有稳定的真机运行证据，所以我没有把它列为已完成的演示功能。它不阻塞当前 P0 主流程。

### 3. 老师要求看代码

优先打开以下三类代码，不要在大量文件中搜索：

- 诊断规则和裁决逻辑：说明匹配、优先级和通用降级。
- Report/History ViewModel：说明页面状态、收藏即时反馈和失败回滚。
- Repository/Storage：说明数据库读写、去重和容量策略。

### 4. 演示突然失败

处理原则：

1. 只重试一次。
2. 仍失败就说明失败位置和错误提示。
3. 使用已有历史报告继续演示后半段流程。
4. 不现场改代码、不反复重启、不把失败隐藏成成功。

## 六、演示中不要说错的内容

- 不要说“使用优先队列匹配错误”，实际是候选规则收集后排序裁决。
- 不要笼统说“稳定排序”，应说“按优先级降序、规则编号升序的确定性排序”。
- 不要把本地关键字接口称为已经训练完成的大模型。
- 不要说“在线 AI 失败一定自动降级”，配置在线 Key 后失败会显示错误，需要手动切回本地能力。
- 不要说“原始日志永远只在内存”，用户主动保存报告后原文会持久化到本地数据库。
- 不要把 OCR、Form Kit、Natural Language Kit、Data Augmentation Kit 或服务卡片的静态检查说成真机运行成功。
- 不要把测试用例数量直接等同于产品质量；测试数量只能作为覆盖证据，还需要说明测试对象和设备验收边界。
- 不要声称已经达到生产级准确率、完成所有设备适配或实现字段级加密。

## 七、最终一句话收尾

> CodePulse 当前最有价值的部分，是在不依赖网络的前提下，把鸿蒙开发错误日志转换成可解释、可执行、可保存和可回看的诊断报告；后续工作重点是扩大真实错误样本覆盖，并完成 OCR 与系统 Kit 的真机验证。
