# 企业老师答辩问题与参考答案

> 对应项目：开发报错诊断助手  
> 使用方式：先说第一句结论，再根据老师追问补充后面的依据和限制。  
> 回答原则：只把有运行证据的能力说成“运行通过”；编译通过、静态检查和未验证必须明确区分。

## 一、项目价值与需求

### 1. 请用 30 秒介绍项目

这是一个面向 HarmonyOS 学习者的开发报错诊断 App。用户输入 ArkTS、Hvigor 或 OHPM 日志后，系统通过本地关键词和正则规则识别主要错误，生成包含证据、原因、步骤和风险的结构化报告，并支持保存、历史、收藏和删除。核心诊断不依赖网络，AI 和平台能力只作为可降级增强。

### 2. DevEco Studio 已经显示错误，为什么还需要这个 App？

DevEco Studio 负责输出编译和构建信息，本项目负责对长日志进行二次整理。它不会替代 IDE，而是帮助初学者区分根因和 `BUILD FAILED` 等包装信息，并把错误转换为有顺序的排查步骤。

### 3. 与直接把日志发给大模型相比有什么优势？

本地规则结果可复现、可解释、可离线，也不需要上传日志和配置 API Key。大模型适合补充未知问题，但存在网络、费用、隐私和幻觉风险，所以项目采用“本地规则稳定基线＋AI 主动增强”的双轨结构。

### 4. 目标用户和需求是怎么确定的？

需求来自课程要求、HarmonyOS 开发中的真实报错样例，以及我在 ArkTS、Hvigor、OHPM 构建过程中遇到的定位困难。目前没有完成大规模用户调研，这是项目的限制；如果产品化，下一步会用任务完成时间、误判率和用户访谈验证需求。

### 5. 最核心的功能是什么？

最核心的是“完整日志到结构化报告”的本地闭环。收藏、AI、Form Kit 等都可以暂时移除，但文本校验、规则诊断和报告生成不能移除。

### 6. P0 和 P1 如何划分？

P0 是首个稳定基线，包括文本输入、本地诊断、报告、保存和历史。P1 是课程第一版增强交付，包括收藏、AI、Form Kit 等。OCR、NLP 和 Data Augmentation 只有形成设备证据后才能计入完成成果。

## 二、错误匹配算法

### 7. 错误类型是怎么匹配的？

系统遍历版本化规则，每条规则包含关键词、正则、错误类型和优先级。任意关键词或任意正则命中，该规则就进入候选集合；候选规则按 `priority` 降序、`ruleId` 升序排列，第一条作为主要错误，其余作为次要标签。

### 8. 关键词和正则是“与”还是“或”？

当前是“或”。关键词适合明显且固定的错误信号，正则适合错误码、边界和可变格式。只要其中一种信号命中，规则就会进入候选集合；报告会记录是关键词信号、模式信号还是两者同时命中。

### 9. 举一个多规则命中的例子

日志同时包含 `not assignable to type` 和 `BUILD FAILED` 时，会命中 ArkTS 编译规则和 Hvigor 构建规则。ArkTS 规则优先级为 120，Hvigor 为 110，所以主要错误是 ArkTS 编译错误，Hvigor 只作为次要标签。

### 10. 为什么不能只看最后一行 `BUILD FAILED`？

因为它只表示构建任务最终失败，不说明最早根因。真正需要修复的通常是前面的类型错误、依赖解析错误或环境配置错误。系统分析完整日志，并通过更具体规则的优先级覆盖包装错误。

### 11. 这是优先队列吗？

不是。当前实现先收集所有命中规则，再调用数组排序。规则只有 12 条，直接排序结构更简单、测试更直观。只有在规则动态大量加入并持续取最高优先级时，优先队列才更有价值。

### 12. “结果稳定”是什么意思？

这里指相同规则版本和相同输入会得到相同主要规则与次要标签。它依靠 `priority` 和 `ruleId` 形成确定顺序，不是指依赖排序算法保持原始输入顺序的“稳定排序”。

### 13. 两条规则优先级相同时怎么办？

按 `ruleId` 字典序升序裁决。这个次级条件消除了规则加载顺序带来的不确定性，因此调整 JSON 中的规则排列不会改变结果。

### 14. 没有规则命中怎么办？

使用唯一的 `generic_unclassified` 通用兜底规则。它只提供补充上下文、定位第一条错误和最小复现等谨慎建议，不虚构具体根因。

### 15. 正则写错了怎么办？

规则资源加载阶段会校验结构；匹配过程中构造正则发生异常时，会统一转换为 `DIAGNOSIS_FAILED`，不会把底层堆栈直接显示给用户。生产级方案还应在发布前对全部规则执行离线编译校验。

### 16. 如何避免关键词误判？

当前通过更具体的规则、正则边界、优先级和测试集降低误判，但简单关键词仍可能产生假阳性。例如字符串 `1401` 理论上可能因为包含 `401` 命中关键词，这是现有规则的改进点。更严格的做法是数字状态码只保留带单词边界的正则，或要求多个信号联合评分。

### 17. 12 条规则是不是太少？

对于课程项目的 ArkTS、Hvigor、OHPM、HTTP、JSON、网络和权限演示范围足够，但对于企业生产明显不够。我把规则引擎和规则资源分离，后续可以按真实错误样本扩展，并通过回归测试防止旧规则结果漂移。

### 18. 规则如何更新？

当前规则作为应用内 `rawfile` 随版本发布，优点是离线和可控，缺点是更新需要发布新版本。企业化后可以增加签名规则包、版本校验、灰度发布和回滚，但不能直接下载未经验证的正则执行。

### 19. 新增规则如何保证不破坏旧规则？

每条规则都应有正样例、负样例、多规则冲突样例和兜底样例。新增规则后运行完整参数化回归，特别检查主要规则、次要标签和优先级是否发生非预期变化。

### 20. 匹配复杂度是多少？

大致是规则数乘以每条规则的关键词和正则扫描成本，再加命中集合排序，即约 `O(R × S × L + M log M)`。目前 `R` 只有 12，输入上限十万字符，实际规模可控；如果规则库扩大，应建立索引、分阶段匹配并把重计算移出 UI 主线程。

### 21. 是否存在正则 ReDoS 风险？

存在理论风险。当前规则是随应用发布的受信资源，且输入有长度上限，风险较低，但没有通用正则超时机制。企业化时应禁止高回溯表达式、增加静态检测、基准测试和执行超时隔离。

## 三、后端设计

### 22. 为什么需要 Node.js 后端？

核心诊断不需要后端。Node.js 服务用于演示 HTTP 调用、健康检查、Agent 接口、统一 `requestId` 和未来模型适配。后端不可达时，App 仍可完成文本诊断、报告和历史功能。

### 23. 后端是真正的 AI 吗？

本地 Node.js 后端本身是轻量关键词分类服务，不是大语言模型。真实生成式 AI 通过 OpenAI 兼容接口调用；界面会明确区分在线模型和本地规则，避免把固定模板包装成 AI。

### 24. 本地规则和后端规则为什么不一样？

App 本地引擎是正式诊断基线，使用 JSON 规则、关键词、正则和优先级；后端目前只是演示性质的 `if/else` 分类。两套逻辑重复是当前技术债，企业化时应抽取共享规则规范或让后端加载同一版本的规则包。

### 25. 两边结果不一致时以谁为准？

当前以 App 本地结构化报告为准，后端和 AI 只提供补充建议，不覆盖本地主要错误。界面需要显示结果来源和规则版本，避免用户混淆。

### 26. `requestId` 有什么作用？

它用于关联一次请求的客户端提示、服务端日志和异常定位。用户不需要看到完整堆栈，只需提供 `requestId`，开发人员就能在脱敏日志中定位对应请求。

### 27. 后端有认证和限流吗？

当前课堂演示后端没有完整认证、账号体系和限流，不能直接公网部署。生产化必须增加 HTTPS、身份认证、请求大小限制、限流、审计日志和密钥管理。

### 28. `Access-Control-Allow-Origin: *` 安全吗？

对无凭证的本地演示接口方便，但不适合带敏感数据的生产服务。如果上线，应限制可信域名，配合认证、CSRF 策略和 HTTPS，并避免返回敏感响应。

### 29. 后端如何处理异常输入？

请求正文最大 1 MB，文本必须非空且不超过十万字符；非法 JSON 返回 400，正文过大返回 413，缺少 prompt 返回 422，未知路径返回 404。生产环境还需补充结构化 Schema 校验和统一错误码。

## 四、AI 与隐私

### 30. AI 具体怎么用？

用户填写 OpenAI 兼容服务地址、模型 ID 和 API Key，再主动提交问题。应用调用 `/chat/completions`，展示模型补充建议；Agent 模式会在提示中加入任务规划要求。AI 结果不能覆盖本地报告，仍需通过实际构建验证。

### 31. 为什么不全部使用大模型？

大模型依赖网络、额度和第三方服务，并可能产生幻觉；开发日志还可能包含私有路径和代码。本地规则用于已知高频错误，AI 只处理用户主动触发的开放问题，这样兼顾稳定性和扩展性。

### 32. API Key 如何保存？

当前只保存在 `AiBackendService` 的运行内存字段中，不写入 Preferences、数据库或日志。退出或进程重启后需要重新填写。企业化可以接入系统安全凭据存储，但不能明文落盘。

### 33. 在线 AI 失败怎么办？

设置 5 秒连接超时和 30 秒读取超时。无 Key 的本地后端不可达时会回退到离线规则；有 Key 的在线请求失败时，当前代码会明确显示 HTTP 或连接错误，并提示检查 Key、模型、额度和地址，不会自动返回本地答案。用户仍可切回本地模式，核心诊断不受影响。若老师追问，应承认“有 Key 请求自动回退”尚未实现。

### 34. 如何防止 AI 幻觉？

目前通过低温度、结果来源标识、本地报告不被覆盖以及“必须重新构建验证”的提示降低风险，但没有完成正式准确率评测。企业化需要固定评测集、引用证据、结构化输出校验和人工确认门槛。

### 35. 如何防止提示注入？

当前没有完整的提示注入防御，因此不能宣称已经解决。现阶段限制为用户主动提交、不自动执行模型返回的命令、Skill 与网络调用分离。生产化还需要系统指令隔离、输入标记、工具白名单、输出过滤和权限沙箱。

### 36. 日志会上传哪些内容？

只有用户在 AI 页面主动提交的文本会发送到配置的在线服务。本地规则诊断不联网。当前应提醒用户先脱敏；企业化应提供自动检测 Token、Authorization、账号、绝对路径和代码片段的预览确认。

### 37. 原始日志是否保存在本地？

诊断会话未保存时，原始文本只在内存中。用户主动保存报告后，`DiagnosisReport.rawText` 会写入 relationalStore，因为历史报告支持展开原文。这属于敏感数据，用户可以删除或清空；当前数据库未做字段级加密，是上线前需要改进的隐私点。

## 五、数据与状态一致性

### 38. 为什么使用 relationalStore？

报告需要按时间排序、按 ID 去重、更新收藏状态、建立索引和执行容量清理，适合关系型存储。主题、字号等简单键值配置使用 Preferences，避免把所有数据都塞进一种存储。

### 39. 报告主要保存什么？

包括报告 ID、错误类型、次要标签、摘要、证据、原因、步骤、建议、风险、原始文本、来源、创建和保存时间、收藏状态以及规则引擎版本。

### 40. 如何避免重复保存？

Repository 保存前按报告 ID 查询；已存在时直接返回原记录。数据库表的 `id` 还是主键，形成业务层和数据库层双重去重。

### 41. 收藏为什么使用即时反馈和回滚？

即时切换可以降低操作延迟感；页面保存旧值后进入 loading，Repository 更新成功则确认，失败则恢复旧值。更新期间禁止再次触发，避免同一页面上的连续点击竞争。

### 42. 是否完全解决并发收藏？

解决了单个页面实例内的重复点击，但没有实现跨页面实例或跨进程的全局锁。企业化可以加入版本号或更新时间，使用乐观锁解决多个消费者同时修改的问题。

### 43. 删除为什么不能先从页面移除？

删除属于不可逆操作。当前先等待数据库结果，成功后刷新列表；失败时保留卡片并提示重试，避免 UI 与数据库不一致。

### 44. 为什么限制 200 条？

这是课程第一版控制本地空间、查询时间和展示规模的产品约束，不是经过大规模压测得出的极限。达到上限时从最早未收藏记录开始清理。

### 45. 如果 200 条全部收藏了怎么办？

Repository 不会自动删除收藏记录，而是返回 `HISTORY_LIMIT_PROTECTED`，提示用户先手动清理收藏。需要注意，当前“插入新报告”和“执行容量清理”不是同一个事务，失败时可能暂时超过上限；这是生产化需要用事务或插入前预检修复的技术债。

### 46. 应用在保存过程中被杀死会怎样？

单条数据库插入本身具有原子性，但插入和后续容量清理不是同一事务，所以可能出现报告已保存、清理尚未完成的状态。下次加载仍能读取报告，但严格一致性需要把保存和淘汰放进同一事务。

### 47. 数据库如何升级？

当前是 V1 Schema，首次建表和索引在事务中执行，失败会回滚。现在只接受版本 0 或当前版本 1，没有实现 V1 到 V2 的实际迁移；新增字段前必须补充显式迁移脚本、备份和回滚测试。

### 48. 数据损坏如何处理？

RowMapper 会校验必填字段、类型、来源枚举和 JSON 数组。解析失败转换为 `DATA_PARSE_FAILED`，页面提示记录损坏并允许删除，不把异常堆栈或损坏原文写入日志。

## 六、HarmonyOS 与平台能力

### 49. 为什么使用 ArkTS Stage 模型？

Stage 模型是当前 HarmonyOS 应用常见工程模式，Ability、窗口、路由、FormExtensionAbility 和生命周期边界更清晰，也符合课程使用 ArkTS 与 DevEco Studio 的要求。

### 50. 为什么选择 API 26？

当前开发环境、SDK、模拟器和工程均基于 API 26，ArkTS、ArkUI、NetworkKit、relationalStore 和 Form Kit 能完成编译。兼容范围不能只根据类型声明推断，仍需在目标真机和模拟器进行运行验证。

### 51. 为什么路由只传 ID？

完整日志和报告对象体积大且包含敏感信息，放入路由参数会增加序列化、泄露和生命周期恢复风险。路由只传 `sessionId`、`reportId` 和受控枚举，页面再从会话或 Repository 加载数据。

### 52. Form Kit 完成到什么程度？

已完成 `2*4` ArkTS 动态卡片、最近报告脱敏快照、Form ID 生命周期、保存删除后的刷新协调、API 26 编译和 348 项 Local Test。模拟器桌面添加、实际刷新、点击拉起、签名 HAP 和真机兼容仍是 `UNVERIFIED`，所以不能说完整运行通过。

### 53. 小组件为什么不显示完整日志？

桌面卡片比应用内页面更容易被旁观者看到，而且空间有限。卡片只保存错误类型、最多 60 字摘要和时间，不保存报告 ID、原文、证据、图片 URI、路径或 Token。

### 54. 小组件如何刷新？

保存、删除和清空完成后，协调服务重新读取最近一条报告，写入独立 Preferences 快照，再尽力刷新已登记的 Form ID。刷新失败不会回滚报告操作，因为卡片只是数据消费者。

### 55. OCR 为什么不作为完成成果？

SDK 静态存在性和部分探针已经检查，但当前没有形成稳定的真实识别结果和完整设备证据。应用保留手动输入降级，所以 OCR 不阻塞 P0，也不能在答辩中说已经完整支持。

### 56. 多窗失败怎么办？

窗口能力由独立 Runtime 封装，创建失败时返回页面内状态提示，不影响诊断数据和主页面。平台能力不能成为本地诊断的单点故障。

## 七、测试与工程质量

### 57. 330 项和 348 项测试为什么不同？

330 项是课程报告和 PPT 使用的 P0 稳定基线。后续加入 Form Kit 增量后，完整 Local Test 扩展到 348 项。答辩先按 PPT 说 330 项；老师追问最新进度时，再说明 Form Kit 增量回归为 348 项，不能把两个阶段混成一次测试结果。

### 58. 测试主要覆盖什么？

覆盖输入边界和 Unicode、规则优先级、多命中与兜底、报告字段、路由白名单、重复保存、最近记录、200 条上限、收藏回滚、删除清空、损坏数据、主题恢复、Form 快照和后端异常。

### 59. 覆盖率为什么只有约 65%？

主要覆盖纯业务逻辑和数据策略；ArkUI 渲染、系统 Kit、Ability 生命周期以及部分设备交互难以由 Local Test 完整覆盖。覆盖率是补充指标，不能替代关键业务路径和真实设备验收。

### 60. 330 个测试多就代表质量好吗？

不代表。数量可能来自参数化样例，真正重要的是是否覆盖根因风险、失败路径和状态一致性。项目重点验证规则冲突、保存去重、收藏回滚、删除失败和资源损坏，而不只是页面能打开。

### 61. 哪些是实际运行证据？

ArkTS 编译、Debug HAP 构建、Local Test 和后端测试有实际命令证据；用户也已在模拟器运行当前 App。Form Kit 的桌面宿主和真机、OCR 真实识别、平板和最终无障碍验收仍需单独区分，不能写成支持。

### 62. Debug HAP 构建成功等于可上线吗？

不等于。它只能证明编译和打包链路通过。上线还需要正式签名、目标设备测试、权限审核、安全测试、性能压测、隐私合规、崩溃监控和发布流程。

### 63. 是否有 CI？

当前主要在本机使用 Hvigor、Local Test、Linter 和 Node 测试验证，没有完整云端 CI。企业化后应在每次合并时自动执行依赖安装、静态检查、单元测试、HAP 构建、敏感信息扫描和产物归档。

### 64. 十万字符日志会不会卡顿？

目前规则少，课程规模下可控，但没有完成正式设备性能基准。因为正则扫描可能占用 UI 线程，企业化应把大日志诊断放到任务线程，增加分段解析、超时、取消和耗时指标。

## 八、安全、日志与上线差距

### 65. 哪些内容禁止写入日志？

完整原始日志、Token、Authorization、API Key、账号、图片 URI、本机绝对路径、签名材料和数据库正文。日志只记录事件名、阶段、稳定错误码、记录数量和 `requestId` 等白名单元数据。

### 66. 如何定位线上异常？

客户端和后端使用结构化事件与 `requestId` 关联，页面只展示简短提示。开发人员通过阶段、错误码、规则版本和请求 ID 定位问题，而不是要求用户上传完整敏感日志。

### 67. 数据库是否加密？

当前没有字段级加密，依赖应用沙箱保护。由于保存报告包含 `rawText`，生产化前应提供“不保存原文”选项、敏感字段加密、安全删除和数据保留期限。

### 68. 当前距离企业上线还缺什么？

主要缺少正式后端认证与部署、规则灰度更新、真实用户评测、性能压测、数据库加密、AI 安全评估、完整 CI/CD、正式签名、手机和平板兼容、无障碍以及真机平台能力验收。

## 九、个人能力与项目管理

### 69. 使用大模型开发，如何证明自己理解代码？

我能够解释关键调用链、数据模型、失败路径和未完成边界，也通过构建、测试和实际运行验证生成结果。大模型用于提高编码和文档效率，最终设计取舍、代码审查、错误修正和验收责任仍由我承担。

### 70. 大模型生成错误代码时怎么发现？

先用编译器和类型系统发现结构错误，再用单元测试验证业务边界，用 Linter 和 `git diff` 检查质量，最后在模拟器验证页面和生命周期。对于不确定的 HarmonyOS Kit，不根据名称猜测，必须检查 SDK 声明、最小编译和设备证据。

### 71. 项目最难的部分是什么？

不是单独做一个页面，而是保证规则结果确定、异步状态和数据库一致，以及增强能力失败时不破坏主流程。例如收藏需要即时反馈和失败回滚，保存与容量限制还涉及事务边界。

### 72. 当前最大的技术债是什么？

本地和后端规则重复、保存与容量清理未处于同一事务、规则关键词仍可能误判、保存原文未加密、AI 缺少正式评测与提示注入防御，以及部分平台能力缺少设备闭环。

### 73. 如果再给一周，优先做什么？

先完成设备与安全闭环，而不是继续堆功能：修复规则误判、把保存和容量清理事务化、完成 Form Kit 与 OCR 真机验证、增加性能基准和敏感数据保护，再补 CI。

### 74. 如果三个人继续开发，如何分工？

一人负责规则引擎、规则样本与评测；一人负责 ArkUI、设备适配和 Form/OCR 平台能力；一人负责存储、后端、AI 安全和自动化流水线。三人通过统一模型、接口契约和测试基线协作。

## 十、现场最危险的追问

### 75. 你的项目有没有你现在解决不了的问题？

有。Form Kit 和 OCR 的设备闭环、保存报告原文的加密、AI 提示注入防御、规则大规模准确率评测都没有完全解决。我已经提供降级路径并在文档中标记，下一步应优先验证这些风险，而不是把它们包装成已完成。

### 76. 如果老师指出代码存在缺陷，应该怎么回答？

先确认具体场景，再说明当前行为、影响范围和最小修复。例如“200 条全部收藏后保存可能暂时超过上限”确实是事务边界问题，修复方式是保存前预检，或把插入和淘汰放进同一事务。不要用“测试都通过”回避有效问题。

### 77. 你最希望老师记住项目的哪一点？

它不是把日志简单交给 AI，而是建立了一个离线、可解释、可验证的诊断基线，并对网络、数据库和平台能力失败提供明确降级。

## 十一、回答时不要说错的口径

- 不说“使用优先队列”，应说“收集命中规则后进行确定性优先级排序”。
- 不说“稳定排序”，应说“相同输入和规则版本得到可复现结果”。
- 不说“后端就是 AI”，应区分轻量规则后端和 OpenAI 兼容在线模型。
- 不说“所有在线 AI 失败都会自动返回本地结果”：当前有 Key 请求失败会显示错误，用户可以切回本地模式；无 Key 的本地后端失败才自动调用离线规则。
- 不说“原始日志完全不落盘”，应说“未保存时只在内存，主动保存报告后 `rawText` 会进入数据库”。
- 不说“Form Kit 已运行通过”，应说“代码、编译和测试通过，桌面与真机未验证”。
- 不说“OCR 已完成”，应说“仍缺稳定设备识别证据，手动输入是降级方案”。
- 不把 330 和 348 说成同一次测试：330 是报告基线，348 是 Form Kit 增量后的完整 Local Test。
- 不说“覆盖率证明没有 Bug”，覆盖率只说明执行范围，不代表需求和断言正确。
- 不说“使用大模型所以开发没有风险”，应强调编译、测试、代码审查和设备验证。

## 十二、万能回答结构

遇到没有准备的问题，可以按下面四步回答：

1. **先给结论：**“当前实现支持 / 不支持 / 部分支持。”
2. **说明依据：**指出代码、测试、构建或设备证据。
3. **主动说边界：**说明仍未覆盖的场景和风险。
4. **给改进方案：**说明企业化后的最小修复，不要空泛地说“以后优化”。

示例：

> 当前 Form Kit 属于部分完成。代码、API 26 编译和 348 项 Local Test 有证据，但桌面添加、刷新、点击拉起和真机仍未验证，所以不能说完整支持。下一步需要配置可安装签名，并在目标设备完成添加、冷启动、刷新和隐私验收。
