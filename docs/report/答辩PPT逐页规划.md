# “开发报错诊断助手”答辩 PPT 逐页规划

> 版本：14 页均衡版  
> 依据：`docs/report/课设报告.md`、`docs/report/课设报告撰写要求.md`、现有需求设计文档和当前代码实现  
> 建议时长：7～8 分钟，不含现场 App 操作  
> 规划原则：内容与课设报告一致；核心功能分别独立成页；未完成或未形成设备证据的能力不作为成果展示。

## 1. 整体叙事

PPT 不按照需求文档逐章复述，而是围绕一条可演示、可解释的产品主线展开：

```mermaid
flowchart LR
    A[开发报错难定位] --> B[本地诊断方案]
    B --> C[分层架构]
    C --> D[输入与校验]
    D --> E[规则诊断]
    E --> F[结构化报告]
    F --> G[保存与历史复用]
    G --> H[视觉和增强能力]
    H --> I[工程质量与测试]
    I --> J[成果与总结]
```

四个核心功能必须分别占用一页：

1. 文本输入与校验；
2. 本地规则诊断；
3. 结构化诊断报告；
4. 保存、历史、收藏与删除。

## 2. 视觉设计规范

### 2.1 设计方向

PPT 延续 App 的“莫兰迪语义配色＋发光玻璃”视觉语言，不直接照搬手机页面，而是把页面中的色彩、圆角卡片、柔和阴影和状态语义转换为适合 16:9 大屏展示的版式。

- 主体风格：克制、安静、技术感，不使用高饱和彩虹渐变；
- 视觉母题：半透明圆角玻璃卡片；
- 页面结构：封面和结尾使用深色背景，正文使用浅色背景，形成“深—浅—深”结构；
- 视觉权重：灰蓝占约 65%，鼠尾草绿与陶土红只用于语义强调；
- 每页至少包含一项视觉内容：App 截图、流程图、架构图、数据数字、状态卡片或代码/日志片段；
- 不使用标题下方装饰横线，不堆放大段正文，不把手机截图原比例缩得过小。

### 2.2 色彩令牌

| 用途 | 色值 | 使用方式 |
| --- | --- | --- |
| 浅色背景 | `#ECEEF2` | 主要正文页背景 |
| 浅色卡片 | `#F8F9FB` | 内容卡片、截图承载面 |
| 深色背景 | `#101216` | 封面、结尾、章节过渡 |
| 深色卡片 | `#1A1D23` | 深色页玻璃卡片 |
| 灰蓝主色 | `#7189A6` | 主流程、重点功能、主要数字 |
| 鼠尾草绿 | `#7F9B86` | 收藏、成功、稳定、已完成 |
| 陶土红 | `#B06F61` | 删除、风险、失败和限制 |
| 雾灰蓝 | `#A7B2BE` | 次要流程、边框、辅助信息 |
| 主文字 | `#171A1F` | 浅色页标题和正文 |
| 次要文字 | `#5D6673` | 说明、图注和脚注 |
| 深色页文字 | `#F3F5F8` | 深色页标题与正文 |

### 2.3 字体与排版

- 中文首选：HarmonyOS Sans SC；PowerPoint 环境不可用时使用微软雅黑；
- 英文、数字和代码：HarmonyOS Sans / Aptos / Consolas；
- 页面标题：36～42 pt，粗体；
- 核心数字：48～64 pt，粗体；
- 卡片标题：22～26 pt；
- 正文：18～22 pt；
- 图注和来源：12～14 pt；
- 左右安全边距不小于 0.6 英寸，卡片之间保持 0.3～0.45 英寸间距；
- 每页正文控制在 3～5 个信息点，每个信息点尽量不超过两行。

### 2.4 玻璃效果实现建议

- 卡片圆角统一为 16～22 pt；
- 浅色页卡片使用 90%～96% 不透明白色，深色页使用 70%～85% 不透明深灰；
- 边框使用 1 pt 白色或雾灰蓝，并降低透明度；
- 阴影采用低透明度灰蓝，不使用纯黑硬阴影；
- 重点卡片可以增加灰蓝外发光，但同一页只设置一个最强焦点；
- 鼠尾草绿只表示收藏、成功和稳定，陶土红只表示删除、风险和未完成边界；
- 动画只使用淡入、擦除或 Morph，避免旋转、弹跳和连续闪烁。

HarmonyOS 报错日志诊断工具

- 组别：待填写；
- 学号：待填写；
- 姓名：待填写。

**核心文案**

> 从复杂日志到可执行排查步骤

**版式**

- 深色 `#101216` 全屏背景；
- 左侧约 55% 放项目名称、副标题和个人信息；

### 第 2 页：为什么要做这个 App

**核心结论**

初学者面对 ArkTS、Hvigor 和 OHPM 长日志时，常被末尾包装错误误导，真正需要的是定位最早根因和获得有顺序的操作建议。

**页面内容**

- 痛点一：原始日志长，错误被多层任务信息包裹；
- 痛点二：错误信息常来自终端或聊天截图，不便整理；
- 痛点三：浏览器搜索依赖网络，答案质量不稳定；
- 项目价值：把日志转换为“分类—证据—原因—步骤—风险”。

**版式**

- 浅色背景，左右对比布局；
- 左侧放经过脱敏的 Hvigor/ArkTS 错误日志局部，末尾 `BUILD FAILED` 用陶土红标记；
- 右侧放结构化报告卡片，用灰蓝箭头连接；
- 页面底部放一句价值总结：“少看包装信息，先找可验证根因”。

**讲述重点**

不要泛泛讲“AI 很流行”，直接从自己开发 HarmonyOS 时遇到的错误定位困难切入。

**对应报告**：1.1、2.1。

### 第 3 页：产品目标与完成范围

**核心结论**

项目首先完成不依赖网络的 P0 稳定闭环，再叠加 AI、Agent、Skill、多窗和传感器体验。

**页面内容**

主流程使用 5 个横向节点：

1. 输入日志；
2. 校验与规则匹配；
3. 生成结构化报告；
4. 主动保存；
5. 历史复用。

右下角使用两组状态标签：

- 已完成：文本诊断、报告、保存、历史、收藏、删除、设置、AI/Agent/Skill；
- 不作为完成成果：OCR、Form Kit 桌面闭环、NLP、Data Augmentation Kit。

**版式**

- 横向流程占页面上方 60%；
- 下方左侧用鼠尾草绿卡片表示稳定基线；
- 下方右侧用雾灰蓝卡片表示增强链路；
- 未完成边界使用小号陶土红提示框，不抢占视觉中心。

**讲述重点**

强调“增强能力失败不会阻塞本地诊断”，同时主动说明未完成边界，避免老师追问时口径不一致。

**对应报告**：2.2、2.3、3.5。

### 第 4 页：系统总体架构

**核心结论**

项目采用页面—ViewModel—领域服务—Repository—Storage 分层，页面不直接处理数据库、网络和复杂规则。

**页面内容**

- 页面与组件：展示状态、接收操作；
- ViewModel：编排诊断、报告和历史流程；
- 领域服务：输入校验、规则匹配、报告生成；
- Repository：保存去重、分页、容量与收藏策略；
- Storage/Service：relationalStore、Preferences、rawfile、AI、多窗、传感器；
- 公共基础：`AppResult`、`AppError`、脱敏日志和路由。

**版式**

- 使用五层横向架构图，不使用大段目录树；
- 主依赖箭头统一为灰蓝色；
- 数据存储层用鼠尾草绿，异常和日志边界用陶土红小标签；
- 右侧增加“可测试、可替换、可降级”三个竖向结果标签。

**讲述重点**

选一条调用链说明：诊断页只调用 ViewModel，规则服务生成报告，保存时才经过 Repository 写入数据库。

**对应报告**：3.1。

### 第 5 页：重点功能一——文本输入与校验

**核心结论**

诊断入口先保证输入可控、失败可恢复，再进入规则分析。

**页面内容**

- 支持粘贴、编辑、清空和示例填充；
- 去除首尾空白；
- 检查空输入和 100000 个 UTF-16 单元上限；
- 校验失败保留原文并就近提示；
- 诊断中禁止重复提交；
- 返回修改时从会话恢复原输入。

**版式**

- 左侧 42% 放文本输入页截图；
- 右侧放“输入—校验—会话—诊断”四步流程；
- 在截图上用灰蓝编号标注输入框、提示区和开始诊断按钮；
- 错误提示使用陶土红，但保持低饱和。

**建议现场演示**

先提交空输入展示提示，再填入固定 ArkTS 错误样例进入诊断。

**讲述重点**

强调页面不执行诊断规则，只把输入事件交给 `DiagnosisViewModel` 和 `InputValidationService`。

**对应报告**：3.2、图 3.2。

### 第 6 页：重点功能二——本地规则诊断

**核心结论**

12 条版本化规则通过关键词、正则、优先级和通用兜底实现稳定、离线、可解释的诊断。

**页面内容**

- 输入：经过校验的错误文本；
- 匹配：关键词与正则，不区分大小写；
- 裁决：优先级降序、规则编号升序；
- 输出：主错误类型＋次要标签；
- 兜底：无专用命中时返回谨慎的通用报告；
- 覆盖：ArkTS、Hvigor、OHPM、JSON、HTTP、网络、权限和空值等。

**版式**

- 中央放一条六步规则流水线；
- 左下角放简短脱敏错误样例；
- 右下角放诊断结果标签，不放完整代码；
- 用鼠尾草绿标记“断网可用、结果稳定”，用陶土红标记“未知错误不武断判断”。

**建议动画**

按流水线顺序淡入：文本 → 信号 → 排序 → 主规则 → 标签 → 报告。

**讲述重点**

解释为什么不能只匹配最后一行 `BUILD FAILED`，以及多规则命中时如何保证结果稳定。

**对应报告**：3.2、4.2。

### 第 7 页：重点功能三——结构化诊断报告

**核心结论**

报告不是重复原日志，而是把错误整理为可以逐项验证的解决路径。

**页面内容**

报告结构按阅读优先级排列：

1. 错误分类与摘要；
2. 命中证据；
3. 可能原因；
4. 排查步骤；
5. 建议与风险提示；
6. 可展开的诊断原文。

同时说明：

- 会话报告通过 `sessionId` 获取；
- 历史报告通过 `reportId` 获取；
- 两种来源复用同一个报告页和 ViewModel。

**版式**

- 右侧约 45% 放完整报告页截图；
- 左侧使用六个小卡片解释报告结构；
- 风险提示使用陶土红边框，不使用纯红整块背景；
- 页面中央放一句大字：“从错误描述到可执行步骤”。

**建议现场演示**

展开诊断原文，说明结构化结论与原始日志可以相互核对。

**对应报告**：3.2、图 3.2。

### 第 8 页：重点功能四——历史、收藏与删除

**核心结论**

报告只有在用户主动保存后持久化；收藏、删除和清空均保证界面状态与数据库结果一致。

**页面内容**

- 未保存报告只存在当前会话；
- 保存前按报告 ID 去重；
- 最近记录按保存时间倒序；
- 最多保留 200 条，优先淘汰最早未收藏记录；
- 收藏点击后即时切换，持久化失败自动回滚；
- 单条删除和清空全部均需二次确认；
- 数据库失败时不提前从界面移除记录。

**版式**

- 左侧放历史记录页截图；
- 右侧上方放“内存 → 主动保存 → relationalStore”数据流；
- 右侧下方放三张状态卡：收藏使用鼠尾草绿、删除使用陶土红、普通打开使用灰蓝；
- 用一个小型状态箭头解释“即时反馈—成功确认/失败回滚”。

**建议现场演示**

依次演示保存、进入记录、收藏/取消收藏、删除确认；清空全部只展示确认框，不一定真正执行。

**对应报告**：3.3、图 3.3。

### 第 9 页：页面导航与视觉系统

**核心结论**

四入口悬浮 Dock、莫兰迪语义色和玻璃组件共同形成统一、可辨认的交互系统。

**页面内容**

- 鼠尾草绿：收藏和成功；
- 陶土红：删除、风险和确认；

**版式**

- 右上角放浅色/深色主题小对比；

**讲述重点**

### 第 10 页：AI、Agent 与 Skill

**核心结论**

AI 是用户主动使用的增强能力；本地规则始终是无需网络和密钥的稳定基线。

**页面内容**

- OpenAI 兼容接口：地址、模型、API Key 由用户主动填写；
- 本地 Node.js 后端：零第三方依赖，便于课堂演示和断网降级；
- Agent：Planner → Skill Router → 执行 → Verifier；
- Skill：JSON Formatter、Error Triage、Report Writer、Privacy Redactor；
- API Key 仅保存在当前运行内存；
- 网络失败不影响本地诊断。

**版式**

- 左侧放智能页或 Agent 步骤截图；
- 右侧放四段 Agent 流程，并用四个小卡片表示 Skill；
- 页面底部放“在线增强 / 本地稳定基线”双轨结构；
- 不展示真实 API Key、服务账号或 Authorization。

**建议现场演示**

### 第 11 页：HarmonyOS 交互能力

**核心结论**

平台能力围绕诊断场景接入，并为权限、设备和运行差异提供可操作降级。

**页面内容**

- ArkUI 玻璃与光感：用于 Dock、核心按钮和诊断卡片；
- 加速度传感器：驱动互动诊断卡片，点击可替代；
- 多窗：`WindowRuntime` 管理诊断闪控子窗，失败时回到页面内状态；
- 深浅主题：跟随系统并允许手动切换；
- 平台能力失败不阻塞“文本—规则—报告—历史”主流程。

- 鼠尾草绿标记可用路径，雾灰蓝标记降级路径；

**讲述重点**

### 第 12 页：工程质量、安全与异常处理

**核心结论**

**页面内容**

- `AppResult<T>` 和 `AppError` 统一跨层错误；
- 页面区分 IDLE、LOADING、SUCCESS、ERROR、CANCELLED；

- 传感器权限不可用：点击互动卡片；
- 多窗不可用：使用页面内状态提示；
