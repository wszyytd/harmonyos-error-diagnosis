# 开发报错诊断助手——答辩演示手册

> 适用项目：HarmonyOS 开发报错诊断助手  
> 推荐时长：7 分钟标准演示；另附 3 分钟压缩版  
> 核心原则：先证明 P0 完整闭环，再展示增强能力；不现场冒险演示未验证功能

## 1. 演示目标

答辩演示需要让老师在短时间内确认四件事：

1. 这是使用 ArkTS、ArkUI、Stage 模型和 API 26 开发的 HarmonyOS 应用；
2. 应用具备“输入错误—本地诊断—结构化报告—保存—历史复用”的完整主题和业务闭环；
3. 页面、ViewModel、Service、Repository 和 Storage 分层明确，数据和异常处理可靠；
4. 项目在核心功能之外还实现了主题视觉、Agent、Skill、多窗、传感器和本地后端等增强能力。

```mermaid
flowchart LR
    A[项目定位] --> B[文本诊断]
    B --> C[结构化报告]
    C --> D[保存与历史]
    D --> E[收藏与删除]
    E --> F[主题与视觉]
    F --> G[Skill 或多窗]
    G --> H[测试与总结]
```

**演示主路线**

## 2. 现场演示的功能边界

### 2.1 必须演示

- 首页和四个一级导航入口；
- ArkTS 文本错误诊断；
- 结构化报告中的结论、证据、原因、步骤和风险提示；
- 保存报告；
- 历史记录重新打开；
- 收藏与取消收藏即时变化；
- 单条删除和清空全部的二次确认；
- 深浅主题、字号或光效切换；
- 至少一个 Skill 或确定性工具；
- 测试和构建结果。

### 2.2 时间允许再演示

- 本地 Agent 的 Planner、Skill Router 和 Verifier；
- JSON Formatter；
- HTTP 状态码速查；
- 独立诊断子窗；
- 加速度互动卡片或点击降级；
- 本地 Node.js 后端。

### 2.3 不放入必演路线

- Core Vision OCR；
- Form Kit 桌面卡片闭环；
- Natural Language Kit；
- Data Augmentation Kit；
- 依赖真实 API Key 的在线模型。

这些能力没有完整设备证据。若老师询问，应如实说明当前状态和降级方案，不要为了效果把它们演示成已经完成。

## 3. 演示前一天准备

### 3.1 工程与设备

- 使用 DevEco Studio 打开项目根目录；
- 确认选择 `entry` 模块和 API 26 模拟器；
- 手动运行一次应用，确认首页、诊断、报告、记录和设置页面可打开；
- 提前关闭通知、聊天软件弹窗和无关窗口；
- 模拟器缩放调整到文字清晰、页面完整的位置；
- DevEco Studio 和模拟器不要全屏来回切换，减少现场操作失误。

### 3.2 数据准备

演示前建议保留两条历史记录：

1. 一条 ArkTS 编译错误；
2. 一条 Hvigor 或 OHPM 错误。

其中一条设为收藏，另一条不收藏。这样历史页面能直接展示收藏、取消收藏、单条删除和最近记录，不需要现场连续创建多条数据。

### 3.3 固定演示文本

优先使用应用内置“使用示例”。若需要手动粘贴，使用以下文本：

```text
ArkTS:ERROR Example.ets:12:5
Type 'string' is not assignable to type 'number'.
```

备用 Hvigor 文本：

```text
hvigor ERROR: assembleHap failed
Invalid value of DEVECO_SDK_HOME
BUILD FAILED
```

备用 OHPM 文本：

```text
ohpm ERROR 00608005
Fetch Pkg Info Failed: NOTFOUND package
```

### 3.4 测试证据

提前准备三张清晰截图，放入答辩 PPT：

- `330 项通过，Failure 0，Error 0` 的 Local Test 结果；
- `BUILD SUCCESSFUL` 的 Debug HAP 构建结果；
- Node.js 后端 `13 passed` 的测试结果。

不要在截图中出现 API Key、Token、签名密码、账号或本机隐私文件。

## 5. 七分钟标准演示脚本

## 5.1 开场与项目定位（约 30 秒）

### 操作

停留在首页，不急着点击。

### 推荐讲解

> 老师好，我们的项目是“开发报错诊断助手”，是一款使用 ArkTS 和 ArkUI 开发的 HarmonyOS API 26 应用。它主要解决 ArkTS、Hvigor 和 OHPM 原始日志较长、初学者难以定位根因的问题。核心流程完全在本地运行，即使没有网络，也能完成输入、诊断、报告生成和历史复用。

### 画面重点

- 应用名称和本地优先说明；
- 文本诊断入口；
- 最近诊断记录；
- 首页、智能、记录、设置四个 Dock 入口；
- 灰蓝、鼠尾草绿、陶土红和发光玻璃视觉。

## 5.2 文本诊断（约 90 秒）

### 操作

1. 点击“文本诊断”；
2. 点击“使用示例”，选择 ArkTS 类型错误；
3. 指一下文本编辑器中的文件位置和类型错误；
4. 点击“开始诊断”；
5. 等待进入诊断报告页。

### 推荐讲解

> 诊断页支持输入、粘贴和内置测试用例。点击诊断后，ViewModel 先调用输入校验服务，检查空输入和长度边界，再加载随应用发布的版本化规则。规则引擎同时检查关键词和正则，按照优先级稳定排序，最后生成结构化报告。页面本身不直接处理规则、数据库和网络请求。

### 不要做

- 不要现场输入一大段未经测试的随机日志；
- 不要连续点击“开始诊断”；
- 不要从截图诊断入口开始主流程。

## 5.3 结构化报告（约 60 秒）

### 操作

在报告页从上向下缓慢滚动，依次展示：

1. 错误类型和摘要；
2. 命中证据；
3. 可能原因；
4. 排查步骤；
5. 风险提示；
6. 诊断原文。

### 推荐讲解

> 系统没有只返回一个错误标签，而是按“证据—原因—步骤—风险”组织结果。当前示例被识别为 ArkTS 编译错误，证据来自稳定规则信号。排查步骤会提示先查看第一条编译错误，核对类型声明，再进行最小修改。风险提示强调不能通过关闭严格类型检查或使用宽泛类型掩盖问题。

### 老师可能关注

若老师问“这是不是简单关键词匹配”，回答：

> 关键词只是一个信号，系统还支持正则、多规则命中、优先级裁决、次要标签和通用兜底。规则匹配与报告生成是两个独立模块，并通过单元测试验证稳定排序和损坏规则等边界。

## 5.4 保存、历史与收藏（约 90 秒）

### 操作

1. 点击“保存报告”；
2. 指出按钮变为“已保存”；
3. 点击“收藏报告”；
4. 指出按钮立即变为“取消收藏”；
5. 点击底部“记录”；
6. 找到刚保存的报告；
7. 点击“打开报告”，证明数据可以重新读取；
8. 返回记录页，再点击收藏或取消收藏一次。

### 推荐讲解

> 报告只有在用户主动保存后才写入 relationalStore。保存按报告 ID 去重，防止重复点击产生副本。收藏采用即时反馈和失败回滚：点击后界面立即变化，Repository 更新失败才恢复旧状态，所以不需要切换页面才能看到结果。首页最近记录、报告页和历史页都通过 Repository 获取数据，不共享可变数据库对象。

### 数据策略补充

若老师追问容量：

> 当前最多保留 200 条，超限时优先淘汰最早且未收藏的记录，收藏内容不会被自动删除。

## 5.5 删除和清空确认（约 40 秒）

### 操作

1. 在历史卡片点击“删除”；
2. 展示二次确认对话框；
3. 第一次点击取消，说明取消不会改变数据；
4. 再次点击删除并确认，删除一条备用记录；
5. 点击“清空全部历史记录”，只展示确认对话框后取消，不要清空刚生成的演示记录。

### 推荐讲解

> 删除和清空属于危险操作，必须二次确认。只有数据库真正返回成功后，ViewModel 才从列表移除卡片；如果存储失败，界面保持原状并允许重试，避免假成功。

## 5.6 设置与视觉设计（约 45 秒）

### 操作

1. 点击底部“设置”；
2. 在“浅色”和“深色”之间切换一次；
3. 选择“较大”字号；
4. 在“完整、降低、关闭”光效之间切换；
5. 恢复到最适合当前演示的状态；
6. 打开“隐私与本地数据说明”，展示后立即返回。

### 推荐讲解

> 系统支持跟随系统、浅色和深色主题，字号和光效均可调整并通过 Preferences 保存。按钮使用固定莫兰迪语义配色：灰蓝表示主要操作，鼠尾草绿表示收藏，陶土红表示删除。关闭光效后仍保留颜色、边框和文字，不依赖动画表达状态。

## 5.7 增强能力二选一（约 60 秒）

### 方案 A：Skill，最稳定，优先推荐

1. 进入底部“智能”；
2. 切换到“Skill”；
3. 点击 `JSON Formatter`；
4. 点击 `Error Triage` 或 `Privacy Redactor`；
5. 展示返回结果。

讲解：

> Skill 中心包含 JSON Formatter、Error Triage、Report Writer 和 Privacy Redactor。每个 Skill 只承担一个职责，使用统一输入输出，可以被本地 Agent 编排，不依赖外部网络。

### 方案 B：多窗，有设备效果时使用

1. 进入“多窗”；
2. 点击“打开独立诊断窗口”；
3. 展示子窗内容；
4. 关闭子窗，证明生命周期被释放。

讲解：

> 项目通过 WindowStage 管理真实应用子窗，并提供页面内状态降级。子窗可以独立创建和释放，不会重复叠加旧窗口。

## 5.8 测试与结束（约 45 秒）

### 操作

切换到 PPT 中准备好的测试截图，不要现场重新构建。

### 推荐讲解

> 项目当前有 41 个单元测试文件，本次验证结果是 330 项全部通过，Failure 和 Error 都为 0；Debug HAP 构建成功；零依赖 Node.js 后端 13 项测试通过。测试覆盖输入边界、规则排序、报告生成、保存去重、历史上限、收藏回滚、删除清空、主题状态和后端异常。

结束语：

> 项目最终形成了一条可离线、可解释、可保存和可复用的鸿蒙开发错误诊断闭环。增强能力失败不会阻塞核心功能，未完成的设备能力也没有被写成已经交付。我的演示结束，谢谢老师。

## 8. 现场故障降级方案

| 故障 | 立即处理 | 讲解口径 |
| --- | --- | --- |
| 模拟器未连接 | 在设备管理器重启；同时先讲 PPT | 环境恢复后补演示核心闭环 |
| 应用启动失败 | 重新 Run；展示构建成功截图 | 不修改代码，不现场重装 SDK |
| 历史记录为空 | 用内置示例诊断并保存一条 | 数据只有主动保存后才持久化 |
| 收藏状态异常 | 返回记录页重试；保留错误画面 | 正常设计应即时更新、失败回滚 |
| 后端不可达 | 切回离线规则诊断 | 核心流程不依赖网络 |
| 在线模型失败 | 不排查密钥，直接跳过 | 在线 AI 是增强能力 |
| 多窗打不开 | 展示页面状态并跳过 | 使用当前页面作为降级 |
| OCR 不可用 | 切换文本诊断 | 当前设备未完成 OCR 闭环，不影响 P0 |
| 页面文字过大 | 设置恢复“标准”字号 | 字号与主题可持久化 |
| 时间不足 | 立即切换三分钟压缩版 | 优先保住诊断、保存、历史和测试 |

## 9. 老师常见问题与建议回答

### 9.1 为什么做这个项目？

> HarmonyOS 初学者经常只看到 `BUILD FAILED` 等包装错误，难以定位第一条有效日志。项目把错误文本整理为可以执行的排查报告，并允许保存和复用，主题集中且适合手机端快速查看。

### 9.2 核心功能需要联网吗？

> 不需要。输入校验、12 条规则、报告生成、历史记录和设置都在本地完成。只有用户主动使用在线 API 时才联网。

### 9.3 为什么不用大模型直接回答？

> 大模型依赖网络、密钥和服务稳定性，输出也可能不一致。本地规则作为稳定基线，可离线、可解释、可测试；大模型只负责增强解释，两者互补。

### 9.4 规则引擎如何处理多个命中？

> 先匹配关键词和正则，再按优先级降序、规则编号升序稳定排序。第一项作为主错误，其余转为次要标签；完全没有命中时使用通用兜底，不武断确定根因。

### 9.5 数据保存在哪里？

> 当前输入和未保存报告在内存会话；主动保存的报告进入 relationalStore；主题、字号和光效进入 Preferences；规则随应用以 rawfile 发布；API Key 仅保存在运行内存。

### 9.6 收藏为什么要即时更新？

> 用户点击后应该立刻看到“收藏”变成“取消收藏”。系统先更新页面状态，再写数据库；写入失败才回滚，同时在操作期间禁止重复点击。

### 9.7 如何保护隐私？

> 报告不自动保存，在线请求必须由用户主动触发，API Key 不落盘。日志禁止记录原始错误文本、图片 URI、Token、Authorization 和本机路径，历史页提供删除与清空入口。

### 9.8 项目用了哪些 HarmonyOS 能力？

> 使用 ArkTS、ArkUI、Stage、Navigation、relationalStore、Preferences、WindowStage、传感器、主题与材质能力。OCR 和 Form Kit 尚未形成完整设备证据，因此不作为已完成成果。

### 9.9 为什么页面不直接访问数据库？

> 页面只负责展示和事件，ViewModel 编排业务，Repository 统一数据策略，Storage 封装平台接口。这样能避免页面过重，也可以使用 Fake Repository 测试失败、回滚和边界状态。

### 9.10 测试做了什么？

> 330 项测试覆盖输入边界、Unicode、规则匹配、稳定排序、通用兜底、报告生成、路由、重复保存、容量限制、收藏回滚、删除清空、数据库损坏、主题和公共组件；后端另有 13 项测试。

### 9.11 当前还有哪些不足？

> OCR 和 Form Kit 仍需支持设备闭环；JSON、HTTP 工具还有边界需求需要补齐；平板、无障碍和最终 P0 全量验收仍需补充。报告中已将这些内容明确列为限制。

## 10. 答辩表达注意事项

- 不要说“所有功能全部完成”，应说“核心 P0 闭环已实现，部分设备能力待验证”；
- 不要说“这是大模型自动诊断”，核心是本地规则，大模型是增强；
- 不要把静态编译通过说成真机运行成功；
- 不要现场展示真实 API Key；
- 每个操作先讲目的，再点击，点击后停顿让老师看清变化；
- 页面滚动速度要慢，重点内容停留 2—3 秒；
- 老师追问代码时，先讲模块职责，再打开关键文件；
- 遇到错误不要慌，按本手册的降级路线继续演示。
