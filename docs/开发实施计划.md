# 开发报错诊断助手 P0 开发实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:test-driven-development` for each implementation task and `superpowers:verification-before-completion` before every completion claim. Execute tasks in order; do not start P1 until P0 acceptance passes.

**目标：**按可独立实现、测试和提交的顺序完成开发报错诊断助手 P0稳定基线。

**架构：**单 HAP、分层模块化单体。页面只绑定 ViewModel状态；纯业务放在 Domain Service；数据通过 Repository访问 Preferences、relationalStore和只读资源；HarmonyOS能力通过 Platform Service隔离并可降级。

**技术栈：**最新版 DevEco Studio、HarmonyOS Stage模型、ArkTS、ArkUI、API 26或以上、Navigation、状态管理 V2（具体装饰器待任务 0验证）、Preferences、relationalStore、HiLog、Local Test、Instrument/UI Test。

## 全局约束

- P0是首个稳定基线；P1是课程第一版增强交付，必须在 P0稳定后集成。
- 使用新建 ArkTS Stage模型工程，不使用 JS项目替代。
- P0核心流程不依赖网络、后端、账号或外部 AI。
- 页面不得直接执行网络请求、关系数据库操作、OCR或复杂诊断规则。
- 原始截图不持久化；用户未主动保存的报告只存在内存。
- 不执行日志中的代码或命令，不把完整日志、Token、密码和图片 URI写入日志。
- 每个任务先建立失败测试或可复现验证，再完成最小实现，最后运行相关测试和构建。
- 每个任务完成后只提交与该任务相关的文件；提交前执行 `git diff --check`。
- 真实工程生成的文件名或 Hvigor任务名如与本文不同，以 API 26新建工程为准，并在任务 0证据中记录差异。

## T0 开发环境与关键能力可行性验证

### 任务目标

在写业务代码前确认 API 26开发链路、Core Vision OCR和 P1平台能力的真实可用性，形成可复核证据和阻塞清单。

### 前置条件

- Windows开发机可运行 DevEco Studio；
- 有课程允许使用的模拟器、真机或远程真机方案；
- 已创建 `develop`分支并保持工作区可回退。

### 涉及目录和文件

- 创建：`docs/evidence/task0/environment.md`
- 创建：`docs/evidence/task0/capability-matrix.md`
- 创建：`docs/evidence/task0/error-samples.md`
- 临时生成：DevEco Studio的最小 ArkTS Stage模型验证工程；正式工程路径在 T1确定
- 不修改：需求、概要和详细设计中的产品范围

### 具体实现内容

1. 在 DevEco Studio“About/关于”记录完整版本、Build号和发布时间。
2. 在 SDK Manager记录 HarmonyOS SDK名称、API Version、补丁版本和安装路径；确认 API不低于 26。
3. 使用新建工程向导创建最小 ArkTS Stage模型应用，记录 `compileSdkVersion`、`compatibleSdkVersion`、`targetSdkVersion`的生成值。
4. 执行依赖同步；记录 `ohpm`版本。若工程有依赖，运行项目约定的 `ohpm install`或 DevEco同步操作，确认无仓库和锁文件错误。
5. 使用 DevEco“Build Hap(s)/APP(s)”构建；再使用生成工程自带 Hvigor Wrapper执行对应 assemble任务。具体命令先从工程任务列表确认，不猜测任务名。
6. 在 API 26手机模拟器启动应用，验证冷启动、前后台切换和系统返回；记录设备名称和系统版本。
7. 在当前 SDK官方文档、API索引和 DevEco代码补全中查找 Core Vision OCR准确包名、类名、最低 API和输入类型；记录“存在/不存在/需额外 Kit”。
8. 查明 OCR所需权限：区分系统图片选择器临时 URI授权、媒体读取权限和 Kit能力申请；将准确权限名与申请时机写入能力矩阵。
9. 使用一张包含英文错误日志的固定图片进行最小 OCR探针验证：只验证 API可导入、可编译、可调用和能返回文本，不构建正式页面。
10. 分别在手机模拟器、可用真机和远程真机记录 OCR结果：支持、不可用、未测试及原因。
11. 验证降级方案：移除/关闭 OCR能力后，最小工程仍可进入手动输入占位流程；OCR不可用不能阻止工程构建与启动。
12. 对 Form Kit、Natural Language Kit、Data Augmentation Kit、沉浸光感分别检查：SDK是否含 API、最低 API、是否需能力申请、模拟器/真机支持、能否编译最小探针。此步骤只形成 P1能力矩阵，不实现功能。
13. 收集真实错误样例：ArkTS、Hvigor、OHPM各至少 3条。每条记录原始文本、产生环境、来源、可复现步骤或可信引用，不用大模型生成文本代替真实样例。
14. 将每一项结论标为 `PASS`、`FAIL`、`UNAVAILABLE`或 `BLOCKED`，并写出下一步。

## 执行与提交纪律

每个任务完成时执行：

1. 运行该任务的最小相关测试；
2. 运行受影响模块的回归测试；
3. 运行静态检查和可适用的 debug构建；
4. 执行 `git diff --check`；
5. 检查 `git status --short`，只暂存该任务文件；
6. 使用任务给出的建议提交信息；
7. 在提交后用 `git show --stat -1`确认范围。

不要把多个未验证任务合并为一个巨大提交，也不要在 T24才首次运行测试。
