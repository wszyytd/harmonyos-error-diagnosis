# 诊断与报告页发光玻璃按钮设计

## 1. 背景与目标

文本诊断、截图诊断和诊断报告页面存在使用系统默认蓝色填充的主要按钮，部分主题下按钮文字对比度不足。此次改动将主要操作统一为首页焦点入口已经采用的发光玻璃视觉，同时保持业务流程、状态管理、导航和按钮层级不变。

## 2. 设计范围

纳入发光玻璃样式的操作：

- 文本诊断和截图诊断共用的“开始诊断”按钮。
- 诊断报告页的“保存报告”按钮。
- 诊断报告页的“编辑原文”按钮。

继续使用低强调样式的操作：

- 使用示例。
- 重新诊断。
- 展开或收起原文。
- 返回首页、返回来源页等导航操作。
- 错误提示中的重试或关闭操作。

不修改 OCR、诊断、报告保存、历史记录、路由或数据模型逻辑。

## 3. 方案选择

采用共享 `GlassActionButton` 组件。组件复用首页焦点入口的语义色与视觉规则，但不在本次改动中重构首页入口，避免扩大回归范围。

未采用的方案：

- 页面内复制样式：改动直接，但会造成诊断页和报告页重复维护。
- 修改全局 Button 主题：影响返回、重试等次要按钮，范围过大。

## 4. 组件职责与接口

建议位置：`entry/src/main/ets/components/common/GlassActionButton.ets`。

组件只负责展示和交互反馈：

- 输入按钮文案、稳定控件 ID、启用状态、加载状态和点击回调。
- 消费 `AppStateStore` 的主题、强调色、字体倍率和视觉效果模式。
- 根据启用和加载状态控制点击派发，避免禁用状态触发操作。
- 提供按下和释放时的轻量缩放反馈；视觉效果关闭时不执行发光动画。

组件禁止承担：

- 诊断、保存或路由业务逻辑。
- ViewModel 状态更新。
- 错误提示和重试策略。
- 页面生命周期管理。

## 5. 视觉规范

视觉参数以首页焦点入口为基准：

- 背景使用 `focusSurface`。
- 边框使用 2 vp 的 `focusBorder`。
- 阴影颜色使用 `focusGlow`。
- 浅色主题且视觉效果开启时使用 `lightFocusMaterial`。
- 深色主题且视觉效果开启时使用 `BlurStyle.Thin`。
- 视觉效果为 `FULL` 时使用较强发光，为 `REDUCED` 时降低阴影半径，为 `OFF` 时关闭模糊、材质、阴影和缩放动画。
- 文字使用 `primaryText`、粗体并随 `contentFontMultiplier` 缩放，确保深浅色主题均具有清晰对比度。
- 最小高度 52 vp，圆角 14 vp，宽度由页面设为 100%。

禁用或加载状态不得只依赖颜色表达：按钮使用明确的加载文案或现有状态文案，并禁止再次点击。禁用状态保留可读文字，不使用低对比度蓝底蓝字组合。

## 6. 页面接入

### 6.1 DiagnosisPage

文本诊断和截图诊断共用同一页面，因此只替换一个“开始诊断”按钮。保留现有控件 ID、加载禁用条件、触摸反馈语义和 `diagnose()` 调用，不改变输入校验与诊断状态流。

### 6.2 ReportPage

将“保存报告”和“编辑原文”替换为共享玻璃按钮：

- 保存按钮继续显示“保存报告”“保存中…”或“已保存”，并沿用保存中及已保存时的禁用规则。
- 编辑按钮只在存在可编辑会话时显示，点击后继续调用现有 `editInput()`。
- 保存失败提示、重新诊断和返回按钮保持现有低强调样式。

## 7. 测试与验收

先建立组件视觉状态的纯模型测试或现有组件测试扩展，再实现组件：

- 深色、浅色和不同强调色下使用正确的语义色。
- `FULL`、`REDUCED`、`OFF` 三种视觉效果模式得到正确的阴影与动效开关。
- 启用状态只派发一次点击；加载和禁用状态不派发点击。
- 文案和稳定控件 ID 能正确传递。

完成后执行：

- 完整 Local Test。
- ArkTS Linter。
- Debug HAP 构建。
- ohosTest HAP 构建。
- `git diff --check`。

设备端由用户手动确认文字清晰度、深浅色效果、按压反馈，以及文本诊断、截图诊断和报告页的实际显示。

## 8. 完成标准

- 三个目标界面不再出现文字难以辨认的默认蓝色主要按钮。
- 主要按钮与首页焦点入口具有一致的半透明、边框和发光语言。
- 次要按钮仍保持低强调层级。
- 诊断、保存、编辑和导航行为无回归。
- 不引入 T19～T24 或其他业务功能。
